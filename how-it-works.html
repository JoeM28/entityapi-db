Architecture Overview

How all the pieces connect
One sentence: An HTTP request hits the Controller, which calls the Service, which calls the Couchbase SDK directly, which reads/writes a document in customer_bucket, and the result travels back the same way.
Entry Point
EntityApiApplication
Boots the Spring context. Runs once on startup. After this line finishes, the server is listening on port 8080.
HTTP Layer
AccountController
Receives HTTP. Routes GET → getById(), POST → upsert(). Returns ResponseEntity with the right status code.
Business + DB
AccountService
Maps API model ↔ document. Calls collection.get() and collection.upsert(). Sets server timestamps.
Config
CouchbaseConfig
Creates the Collection bean at startup. AccountService injects it. One connection shared across all requests.
Data Shape (API)
AccountRecord
Generated from entity-api.yaml. camelCase. Carries @NotNull / @Size / @Pattern. This is what the REST client sends and receives.
Data Shape (DB)
AccountDocument
Plain POJO with @JsonProperty("snake_case"). This is what gets stored in Couchbase. The service maps between this and AccountRecord.
Error Handling
GlobalExceptionHandler
Catches AccountNotFoundException → 404. Catches validation failures → 400 with field-level errors. Returns a consistent ErrorResponse JSON.
Exception
AccountNotFoundException
Thrown by AccountService when collection.get() throws DocumentNotFoundException. Carries the accountId that was not found.
Generated models (AccountRecord, Name, Address, ErrorResponse, ErrorResponseErrorsInner) live in target/generated-sources/openapi/…/model/. They are produced at compile time by the openapi-generator-maven-plugin reading entity-api.yaml. You never edit these files manually.

What it is, what it does, why it exists

EntityApiApplication.java

Entry Point
Code
@SpringBootApplication
public class EntityApiApplication {

    public static void main(String[] args) {
        SpringApplication.run(
            EntityApiApplication.class, args);
    }
}
What it does
@SpringBootApplication is 3 annotations in one:
  @SpringBootConfiguration  → marks this as config class
  @EnableAutoConfiguration  → turns on 200+ auto-configs
                               (Couchbase, Jackson, Tomcat…)
  @ComponentScan            → scans this package and all
                               sub-packages for @Service,
                               @Controller, @Configuration

SpringApplication.run() does:
  1. Starts the Spring IoC container
  2. Runs all auto-configurations
  3. Creates all @Bean objects
  4. Starts embedded Tomcat on port 8080
  5. Returns — server is now accepting requests

CouchbaseConfig.java

Configuration
Code
@Configuration
public class CouchbaseConfig {

  @Bean
  public Collection accountCollection(
      Cluster cluster,
      @Value("${spring.data.couchbase.bucket-name}")
      String bucketName) {

    Bucket bucket = cluster.bucket(bucketName);
    bucket.waitUntilReady(
        Duration.ofSeconds(10));
    return bucket.defaultCollection();
  }
}
What it does
Spring Boot auto-configures a Cluster bean from
application.yml:
  spring.couchbase.connection-string: localhost
  spring.couchbase.username: Administrator
  spring.couchbase.password: password

This config class takes that Cluster and:
  1. Opens the bucket (customer_bucket)
  2. Waits up to 10s for it to be ready
     (Couchbase SDK is async-first — without
      waitUntilReady the first request could
      fail with a "not yet ready" error)
  3. Returns the default Collection

AccountService receives this Collection via
constructor injection. It is created once at
startup and reused for every request.

AccountDocument.java

Couchbase Shape
Code
@JsonIgnoreProperties(ignoreUnknown = true)
public class AccountDocument {

  @JsonProperty("doc_type")
  private String docType;

  @JsonProperty("account_number")
  private Long accountNumber;

  @JsonProperty("name")
  private NameDoc name;

  @JsonProperty("last_update_date")
  private String lastUpdateDate;

  // ... getters / setters

  public static class NameDoc { ... }
  public static class AddressDoc { ... }
}
What it does
This is the data shape stored inside Couchbase.
It is a plain Java object (POJO).

@JsonProperty("snake_case") tells Jackson how
to serialize/deserialize each field:
  Java field   → Couchbase JSON key
  accountNumber → "account_number"
  legalName    → "legal_name"
  countryCode  → "country_code"

The Couchbase SDK serialises this with Jackson
when you call collection.upsert(key, doc)
and deserialises it when you call
result.contentAs(AccountDocument.class).

@JsonIgnoreProperties(ignoreUnknown = true)
is a safety net — if Couchbase returns any
fields not in this class (e.g. "_class" from
old Spring Data documents) Jackson ignores
them instead of throwing an error.

NameDoc and AddressDoc are inner static classes
for the nested objects stored in Couchbase:
  "name": { "legal_name": "...", "dba_name": "..." }
  "address": { "line1": "...", "city": "...", ... }

AccountController.java

HTTP Layer
Code
@RestController
@RequestMapping("/entity")
public class AccountController {

  @GetMapping("/account/{accountId}")
  public ResponseEntity<AccountRecord>
      getAccountById(
        @PathVariable Long accountId) {

    return ResponseEntity.ok(
        accountService.getById(accountId));
  }

  @PostMapping("/account")
  public ResponseEntity<AccountRecord>
      upsertAccount(
        @Valid @RequestBody
        AccountRecord accountRecord) {

    boolean exists = accountService
        .exists(accountRecord
            .getAccountNumber());
    AccountRecord saved =
        accountService.upsert(accountRecord);
    return ResponseEntity
        .status(exists ? 200 : 201)
        .body(saved);
  }
}
What it does
@RestController = @Controller + @ResponseBody
  All return values are written directly as
  JSON to the HTTP response body.

@RequestMapping("/entity")
  Every method in this class is prefixed with
  /entity. So:
    @GetMapping("/account/{accountId}")
    → handles GET /entity/account/123456789

@PathVariable Long accountId
  Spring extracts "123456789" from the URL
  and converts it to a Long automatically.

@Valid @RequestBody AccountRecord
  @RequestBody → Jackson deserializes the
    incoming JSON body into AccountRecord
  @Valid → triggers Jakarta Bean Validation
    on the AccountRecord before the method
    body runs. If @NotNull fields are missing
    or @Pattern fails, Spring throws
    MethodArgumentNotValidException and
    GlobalExceptionHandler returns 400.

ResponseEntity.status(exists ? 200 : 201)
  Returns HTTP 200 if the account already
  existed (update), 201 if it was new (create).
  This matches the API spec exactly.

AccountService.java

Business Logic + DB
Code
@Service
public class AccountService {

  private final Collection collection;

  public AccountService(Collection coll) {
    this.collection = coll;
  }

  public AccountRecord getById(Long id) {
    try {
      GetResult r = collection.get(
          String.valueOf(id),
          GetOptions.getOptions()
              .timeout(Duration.ofMillis(200)));
      return toRecord(
          r.contentAs(AccountDocument.class));
    } catch (DocumentNotFoundException e) {
      throw new AccountNotFoundException(id);
    }
  }

  public AccountRecord upsert(...) {
    AccountDocument doc = toDocument(record);
    collection.upsert(
        String.valueOf(doc.getAccountNumber()),
        doc,
        UpsertOptions.upsertOptions()
            .timeout(Duration.ofMillis(500)));
    return toRecord(doc);
  }
}
What it does
@Service — marks this as a Spring-managed bean.
Spring creates one instance at startup and
injects it into AccountController.

The Collection is injected via constructor.
It was created by CouchbaseConfig at startup.

collection.get(key, options)
  → KV GET on port 11210 (not N1QL)
  → Returns a GetResult
  → result.contentAs(AccountDocument.class)
     deserializes the stored JSON back into
     an AccountDocument using Jackson

collection.upsert(key, doc, options)
  → KV UPSERT on port 11210
  → Inserts if key does not exist
  → Replaces if key already exists
  → doc is serialized to JSON by Jackson
     using the @JsonProperty annotations

toDocument() / toRecord()
  These are private mapping methods that
  convert between the two data shapes:
    AccountRecord  ← API model (camelCase)
    AccountDocument ← DB model (snake_case)
  They also set lastUpdateDate and
  lastUpdateTimestamp on every write.

AccountNotFoundException + GlobalExceptionHandler

Error Handling
AccountNotFoundException
public class AccountNotFoundException
    extends RuntimeException {

  public AccountNotFoundException(Long id){
    super("No Account found with "
        + "accountNumber " + id);
  }
}

--------------------------------------------

@RestControllerAdvice
public class GlobalExceptionHandler {

  @ExceptionHandler(AccountNotFoundException.class)
  public ResponseEntity<ErrorResponse>
      handleNotFound(...) {
    // builds ErrorResponse, returns 404
  }

  @ExceptionHandler(MethodArgumentNotValidException.class)
  public ResponseEntity<ErrorResponse>
      handleValidation(...) {
    // builds ErrorResponse with field errors, 400
  }
}
How exceptions flow
AccountNotFoundException (custom):
  Thrown by AccountService.getById() when
  collection.get() raises DocumentNotFoundException.

  Flow:
    collection.get("999")
    → Couchbase: document not found
    → SDK throws DocumentNotFoundException
    → service catches it
    → throws AccountNotFoundException
    → GlobalExceptionHandler catches it
    → returns 404 JSON:
      {
        "status": 404,
        "error": "Not Found",
        "message": "No Account found with
                    accountNumber 999",
        "path": "/entity/account/999"
      }

MethodArgumentNotValidException (Spring):
  Thrown automatically by Spring when @Valid
  fails on the @RequestBody.

  Flow:
    POST body missing "legalName"
    → @Valid triggers bean validation
    → @NotNull on name.legalName fails
    → Spring throws exception
    → GlobalExceptionHandler catches it
    → returns 400 JSON with errors array:
      { "errors": [
          { "field": "name.legalName",
            "message": "must not be null" }
        ]
      }

GET /entity/account/{accountId} — Step by Step

curl http://localhost:8080/entity/account/123456789
1

HTTP GET request arrives at embedded Tomcat

The request hits the Tomcat server running inside the Spring Boot JVM on port 8080. Tomcat parses the HTTP headers and URL.

GET /entity/account/123456789 HTTP/1.1
Host: localhost:8080
2

Spring MVC routes to AccountController.getAccountById()

Spring's DispatcherServlet matches the URL pattern /entity/account/{accountId} to the @GetMapping method. It extracts "123456789" from the path and converts it to Long accountId = 123456789L.

3

Controller calls accountService.getById(123456789L)

No business logic in the controller — it just delegates to the service immediately.

4

Service calls collection.get("123456789") — KV GET on port 11210

The Couchbase SDK sends a binary protocol GET command directly to port 11210. This is the Key-Value service — not SQL, not HTTP. It is the fastest way to read a document in Couchbase.

collection.get("123456789", GetOptions.getOptions().timeout(Duration.ofMillis(200)))
5a

Happy path — document found

Couchbase returns the stored JSON. The SDK wraps it in a GetResult object.

Couchbase returns raw JSON stored under key "123456789":
{
  "doc_type": "Account",
  "account_number": 123456789,
  "name": { "legal_name": "Acme Corporation", "dba_name": "Acme Corp" },
  "address": { "line1": "123 Main Street", "city": "Dallas",
               "state": "TX", "country_code": "USA" },
  "dob": "01-15-1990",
  "status": "ACTIVE",
  "last_update_date": "2026-03-02",
  "last_update_timestamp": "2026-03-02T14:00:00Z"
}
5b

Unhappy path — document not found → 404

If no document exists for that key, Couchbase SDK throws DocumentNotFoundException. The service catches it, wraps it in AccountNotFoundException, which GlobalExceptionHandler turns into a 404 JSON response.

6

SDK deserializes JSON → AccountDocument via Jackson

result.contentAs(AccountDocument.class)
// Jackson reads "account_number" → accountNumber (Long)
// Jackson reads "legal_name"     → legalName (String)
// Jackson reads "country_code"   → countryCode (String)
7

Service maps AccountDocument → AccountRecord (toRecord)

The mapping method converts from DB shape (snake_case, String status) to API shape (camelCase, StatusEnum).

doc.getStatus()  →  "ACTIVE"  →  StatusEnum.ACTIVE
doc.getLastUpdateDate() → "2026-03-02" → LocalDate.parse("2026-03-02")
8

Controller returns ResponseEntity.ok(accountRecord) — HTTP 200

Jackson serializes AccountRecord to JSON using the @JsonProperty annotations on the generated model. The result is written to the HTTP response body.

HTTP/1.1 200 OK
Content-Type: application/json

{
  "docType": "Account",
  "accountNumber": 123456789,
  "name": { "legalName": "Acme Corporation", "dbaName": "Acme Corp" },
  "address": { "line1": "123 Main Street", "city": "Dallas",
               "state": "TX", "countryCode": "USA" },
  "dob": "01-15-1990",
  "status": "ACTIVE",
  "lastUpdateDate": "2026-03-02",
  "lastUpdateTimestamp": "2026-03-02T14:00:00Z"
}

POST /entity/account — Step by Step

curl -X POST http://localhost:8080/entity/account -d '{…}'
1

HTTP POST arrives with JSON body

POST /entity/account HTTP/1.1
Content-Type: application/json

{
  "docType": "Account",
  "accountNumber": 123456789,
  "name": { "legalName": "Acme Corporation" },
  "address": { "line1": "123 Main Street" },
  "status": "ACTIVE"
}
2

Jackson deserializes JSON body → AccountRecord (@RequestBody)

Spring reads the Content-Type: application/json header and uses Jackson to convert the raw JSON into an AccountRecord object. Enum strings like "ACTIVE" are converted to StatusEnum.ACTIVE via @JsonCreator.

3

@Valid triggers Bean Validation on AccountRecord

Before the controller method body runs, Spring calls the Jakarta Bean Validator on the AccountRecord. Validation rules come from the generated model (they were produced by the openapi-generator reading the YAML constraints).

Validated automatically:
  accountNumber  → @NotNull  (required in spec)
  name           → @NotNull + @Valid (required + validates Name inside)
  name.legalName → @NotNull + @Size(max=20)
  address.state  → @Pattern("^[A-Z]{2}$")
  address.countryCode → @Pattern("^[A-Z]{3}$")
  dob            → @Pattern("^(0[1-9]|1[0-2])-(0[1-9]|…)$")

If any fail → MethodArgumentNotValidException
            → GlobalExceptionHandler → HTTP 400 with errors array
4

Controller checks if account already exists — KV EXISTS

accountService.exists(123456789L)
→ collection.exists("123456789")  port 11210
→ returns true (update) or false (create)
→ stored to decide the response status code later
5

Service converts AccountRecord → AccountDocument (toDocument)

The mapping method also sets the two server-side timestamps:

doc.setLastUpdateDate(LocalDate.now().toString())
   → "2026-03-02"

doc.setLastUpdateTimestamp(Instant.now().toString())
   → "2026-03-02T16:30:00.123Z"

These values come from the server clock — the client never provides them.
6

collection.upsert(key, doc) — KV UPSERT on port 11210

Jackson serializes AccountDocument to JSON using @JsonProperty annotations (camelCase Java → snake_case JSON). The Couchbase SDK stores that JSON under key "123456789".

collection.upsert("123456789", doc, UpsertOptions.upsertOptions().timeout(Duration.ofMillis(500)))
→ If key "123456789" did not exist → INSERT (new document)
→ If key "123456789" existed       → REPLACE (full document overwrite)
7

Service maps AccountDocument → AccountRecord (toRecord) and returns

The returned AccountRecord includes the server-set timestamps so the client can see when the record was last modified.

8

Controller returns 201 (new) or 200 (updated)

existed = false → HTTP 201 Created  (first time this accountNumber is stored)
existed = true  → HTTP 200 OK       (existing document was updated)

How Spring Boot Runs — JVM, Memory, CPU, Port

What actually happens when you hit Run
Spring Boot is not a framework you deploy into a server. It packages an entire HTTP server (Tomcat) inside your JAR. When you run the JAR, you start a JVM process that contains both your application and the server. There is no external Tomcat to configure.

The JVM Process

java -jar entity-api-service-1.0.0.jar
         ↑
         One JVM process. Contains:
           ├── Embedded Apache Tomcat (HTTP server, port 8080)
           ├── Spring IoC Container (all beans: Controller, Service, Config…)
           ├── Couchbase SDK client (persistent connection to localhost:11210)
           ├── Jackson ObjectMapper (JSON serialiser/deserialiser)
           ├── Jakarta Bean Validator
           └── Your compiled code

         The process stays alive until you stop it.
         Every incoming HTTP request is handled by a thread from Tomcat's thread pool.
         The Couchbase connection is shared — one connection, many requests.
ResourceDefault ValueWhere to change
HTTP port8080server.port=9090 in application.yml
JVM heap (max)¼ of your system RAM (e.g. 4 GB system → 1 GB heap)java -Xmx512m -jar … at startup
JVM heap (initial)⅛ of system RAMjava -Xms256m -jar …

In IntelliJ: the JVM process is started by IntelliJ's run configuration. You can see memory usage in the bottom bar of the IDE. Click the memory indicator to trigger a GC. The console tab shows all Spring Boot log output including startup time and port.

Couchbase Docker — Complete Setup

From zero to a working bucket in 5 minutes

Step 1 — Start the container

docker run

Terminal
docker run -d --name couchbase-local \
  -p 8091-8094:8091-8094 \
  -p 11210:11210 \
  couchbase:latest

# Port mapping explained:
#   8091 → Couchbase Web UI (cluster admin)
#   8092 → Views / XDCR
#   8093 → Query Service (N1QL) — needed if you run SELECT queries later
#   8094 → Search Service
#   11210 → Key-Value Service — this is what the SDK uses for get/upsert

Step 2 — Open the Web UI and create the cluster

http://localhost:8091
  • Open http://localhost:8091 in your browser. Wait ~20 seconds after docker run for the server to start.
  • Click "Setup New Cluster"
  • Cluster Name: local-dev
  • Admin Username: Administrator
  • Admin Password: password  (matches application.yml)
  • Click Next: Accept Terms → check the box → Finish With Defaults

Step 3 — Create the bucket

Couchbase UI → Buckets tab
  • Click Buckets in the left navigation
  • Click "ADD BUCKET" (top right)
  • Name: customer_bucket  (must match application.yml exactly)
  • Memory Quota: 256 MB (safe for local Docker)
  • Leave everything else as default → click Add Bucket

Step 4 — Bucket, Scope, and Collection — what to create

Short answer: Create only the bucket. Do not create a scope or collection. The app uses _default scope and _default collection automatically — they already exist inside every new bucket.
ConceptWhat it isWhat to do
Bucket Top-level data container. Like a database. Has a name and a memory quota. Create this: customer_bucket
Scope Namespace inside a bucket (like a schema). Used for multi-tenancy. Use _default — already exists. Do not create one.
Collection Groups of documents inside a scope (like a table). Stores the actual documents. Use _default — already exists. Do not create one.

Where documents are stored

Full path to where AccountDocument lives:

  customer_bucket  →  _default (scope)  →  _default (collection)

In the SDK call:
  cluster.bucket("customer_bucket")
         .defaultCollection()      ← this is _default._default

Document key "123456789" resolves to:
  customer_bucket._default._default["123456789"]  =  { the JSON document }

Looking Up Data in the Couchbase UI

Three ways to find and inspect documents

Method 1 — Document Browser (look up by document key)

Best for: Checking a specific document you just inserted. Direct key lookup — instant.
Couchbase UI → Buckets → customer_bucket → Documents
  1. 1Click Buckets in the left nav
  2. 2Click Documents next to customer_bucket
  3. 3In the Document ID field at the top, type 123456789 and press Enter
  4. 4The document JSON appears. You can edit it directly in the UI and save.
What you will see:
{
  "doc_type": "Account",
  "account_number": 123456789,
  "name": { "legal_name": "Acme Corporation", "dba_name": "Acme Corp" },
  "address": { "line1": "123 Main Street", "city": "Dallas", ... },
  "status": "ACTIVE",
  "last_update_date": "2026-03-02",
  "last_update_timestamp": "2026-03-02T16:30:00.123Z"
}

Method 2 — Query Workbench (N1QL — browse all documents)

Best for: Finding documents by field value, listing all accounts, filtering by status. Requires a primary index — run the CREATE INDEX once below.
Couchbase UI → Query tab

Click Query in the left nav. A SQL-like editor opens. First, create a primary index (only needed once):

-- Run this ONCE to enable N1QL queries on the bucket:
CREATE PRIMARY INDEX ON customer_bucket;

-- List all documents
SELECT * FROM customer_bucket;

-- List all Account documents
SELECT * FROM customer_bucket WHERE doc_type = "Account";

-- Find by status
SELECT * FROM customer_bucket
WHERE doc_type = "Account" AND status = "ACTIVE";

-- Find a specific account
SELECT * FROM customer_bucket
WHERE account_number = 123456789;

-- Count all accounts
SELECT COUNT(*) AS total FROM customer_bucket
WHERE doc_type = "Account";

Method 3 — Browse all documents in the bucket

Couchbase UI → Buckets → customer_bucket → Documents (leave ID blank)
  • Click Buckets → Documents but leave the Document ID field empty
  • Click Retrieve Docs — shows the first 10 documents as JSON cards
  • Click any card to open the full document in an editor
  • Use the page forward/back buttons to browse more documents
  • This view shows all documents regardless of doc_type — both Account and Customer documents (when you add Customer support later) will appear here
Tip: After you POST an account via curl or IntelliJ HTTP client, go straight to Buckets → Documents → enter the accountNumber as the Document ID. You will see the exact JSON stored in Couchbase — including the snake_case field names, and last_update_timestamp set by the server. This is the fastest way to verify your POST actually worked.
