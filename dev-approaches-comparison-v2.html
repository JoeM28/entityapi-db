Entity API DB — V2 — Field Addition Case Study

Markdown-First Spec-Driven DevelopmentNEW IN V2

A deep dive into four development approaches for adding emailConfidenceCode to the Entity API — with the Markdown-First iteration loop as the centrepiece new idea. Is iterating a design markdown with AI before touching the spec a good way to do Spec-Based coding? Yes — and here is exactly how.

The Core Insight

What goes wrong without this pattern

Most developers (and most AI tools) jump straight from a sentence of intent to code. The spec, if it gets updated at all, is updated after the code. Tests are written last. Edge cases are discovered in production.

Even "Spec-Driven Development" has a gap: the spec itself can be wrong. If you commit a bad spec, code gets built on a bad contract. Changing the spec later breaks consumers.

⚠️
The expensive mistake: In entityapi-db, AccountRecord is auto-generated from entity-api.yaml. A field added to the YAML without careful thought becomes a permanent API contract — changing or removing it is a breaking change for any consumer of the API.

The Markdown-First loop fixes this

A Design Markdown is a lightweight, cost-free, deletable artifact. You iterate it with AI until the design is solid. Only then does the spec get updated. The key idea:

💡
Markdown is the thinking space. YAML is the commitment.

You can trash a markdown doc at zero cost. A bad YAML change requires a versioned deprecation cycle. Use markdown to resolve every open question before touching the spec.

AI is an outstanding thought partner at the markdown stage because it can:

  • Identify edge cases you haven't considered
  • Challenge naming and type decisions before they're locked in
  • Generate the full test matrix from the design intent
  • Spot consistency issues with existing fields

Without knowing this project's OpenAPI-first constraint, an AI doing vibe coding will almost certainly edit the generated file directly:

AI vibe output — WRONG: editing the generated source file

⚠️
This is not a hypothetical. In real teams using vibe coding without process, this exact mistake causes hours of debugging: "it worked yesterday, why did it stop?" The answer is always "someone ran mvn compile and wiped the generated-sources edit."

A well-context-prompted AI will produce the correct changes, but the developer must explicitly tell it about the OpenAPI-first constraint every time. With vibe coding, that context is typically missing from the prompt.

3
Edge Cases Not Covered — the invisible debt

Even when vibe coding produces syntactically correct code, the following are typically not handled unless the developer explicitly asks:

What vibe coding leaves unverified
✗ Legacy Couchbase documents (no email_confidence_code field) → NPE on GET
✗ fromValue() throws IllegalArgumentException for unknown strings → 500 in prod
✗ Round-trip fidelity: POST "LOW" → GET returns "LOW" (not tested)
✗ Postman collection NOT updated → integration tests don't cover new field
✗ entity-api.yaml examples not updated → API docs misleading
✗ No unit tests → next developer who refactors AccountService gets no warning

Each item above is a future production incident waiting to happen. There is nothing in the codebase to catch them — no tests, no spec validation, just silence until a user hits the bug.

✅ When it works

  • Developer knows the codebase deeply (knows about generated sources)
  • Throwaway prototype — not going to production
  • Solo project with no downstream API consumers
  • Simple cosmetic change with no business logic

❌ When it fails

  • AI doesn't know about project-specific constraints (OpenAPI-first trap)
  • No regression tests — every future change is a gamble
  • Knowledge lives in developer's head, not in the codebase
  • Legacy document edge cases missed until production NPE
  • Technical debt compounds with each vibe session
  • Code review is impossible without a spec to compare against

# Design: emailConfidenceCode field on AccountRecord

## Context
The email verification service (external) now produces a confidence signal
for each account email address. We need to store and return this signal
via the Entity API so downstream consumers can act on it.

## Field Definition
- Name: emailConfidenceCode
- Type: enum string
- Values: HIGH, MEDIUM, LOW, UNKNOWN
- Required: No (optional)
- Who sets it: The client (email verification service) on upsert

## Couchbase Storage
- Stored as: email_confidence_code (snake_case)
- Location: top-level field on the account document

## API Contract
- Appears in POST /entity/account request body (client sets it)
- Appears in GET /entity/account/{id} response (returned to caller)
- Appears in POST /entity/account response (echo back)

## Open Questions
- ??? What should we return for accounts created before this field existed?
- ??? Should the client be able to set this, or is it server-set only?
- ??? What happens if the client sends a value not in the enum?

## Files to Change
- entity-api.yaml
- AccountDocument.java
- AccountService.java

The developer shares the markdown with the AI and asks it to critique the design, not generate code.

Here is my design doc for adding emailConfidenceCode to the Entity API.

[pastes design-v1.md above]

Review this design. What gaps, risks, or unresolved questions do you see? Don't write any code — just analyse the design.
Good start. Here are the gaps I see:

1. Open Question 1 is critical: Your existing Couchbase documents don't have email_confidence_code. When AccountService.toRecord() calls fromValue(null) it will throw IllegalArgumentException → 500 on every GET for a legacy account. You need to decide: return null, or return UNKNOWN as a default?

2. Open Question 2 matters for your model generation: If the field is client-settable, it goes in the YAML as a normal property. If it's server-set (like lastUpdateDate), it should be readOnly: true and never accepted from the client. Your context says "the email verification service sets it" — but is that service the API client, or a separate internal process?

3. Open Question 3 is already answered by OpenAPI: If you define it as an enum in the YAML, the generated Jakarta validation rejects invalid values with a 400 automatically. No additional service-layer code needed.

4. Missing: What about CustomerRecord? Your spec has both AccountRecord and CustomerRecord. Should CustomerRecord also get this field? Decide now to avoid a second PR later.

5. Missing: Couchbase migration. Will existing documents be backfilled, or left without the field? This affects whether callers can rely on the field always being present.

6. Missing: Test scenarios. The design doc has no test cases. Add them so the implementation phase has a clear acceptance checklist.

# Design: emailConfidenceCode field on AccountRecord
## Status: DRAFT v2 — under review

## Context
The email verification service (external) produces a confidence signal per account.
We store and expose this signal so downstream risk systems can act on it.
The email verification service is an API CLIENT — it sets this field on upsert.

## Field Definition
- API name:        emailConfidenceCode (camelCase, client-settable)
- Couchbase name:  email_confidence_code (snake_case)
- Type:            optional string enum
- Enum values:     HIGH | MEDIUM | LOW | UNKNOWN
- readOnly:        false (client sets it — NOT server-set like lastUpdateDate)
- CustomerRecord:  OUT OF SCOPE for this change — separate ticket ENG-4421

## Legacy Document Handling (RESOLVED)
Decision: Return null (not UNKNOWN) for accounts without the field.
Reason: Callers should distinguish "not yet assessed" (null) from
"assessed and result is unknown" (UNKNOWN). Masking as UNKNOWN loses info.
Implementation: null-guard in toRecord() — if doc field is null, leave
r.emailConfidenceCode as null. No default injection.

## Enum Validation (RESOLVED)
OpenAPI enum constraint + Jakarta validation handles invalid values → 400.
No additional service-layer validation needed.

## Couchbase Migration
No backfill. Existing documents left as-is (no email_confidence_code field).
@JsonIgnoreProperties(ignoreUnknown=true) already present on AccountDocument
— Jackson will simply leave the field null on deserialization.

## Test Scenarios (ADDED)
1. POST with HIGH → response contains HIGH, Couchbase stores HIGH
2. POST with each of MEDIUM, LOW, UNKNOWN → stored and returned correctly
3. POST without the field → 201, field absent/null in response
4. POST with invalid value "SUPER_HIGH" → 400
5. GET account WITH email_confidence_code in Couchbase → returned
6. GET account WITHOUT email_confidence_code (legacy doc) → null, no 500
7. Existing unit tests not broken (regression)

## Files to Change
- entity-api.yaml (AccountRecord schema + POST/GET examples)
- AccountDocument.java (add email_confidence_code field)
- AccountService.java (toDocument + toRecord mapping)
- AccountServiceTest.java (7 new test cases)
- tests/integration-tests.postman_collection.json (2 new requests)

## Open Questions (REMAINING after Round 1)
- ??? Should POST example in YAML include emailConfidenceCode?
- ??? Does the GET response example need updating?

AI Iteration — Round 2: Edge Cases & Consistency Check
Updated design is attached (v2). Two open questions remain about YAML examples. Also — are there any edge cases I'm still missing? Specifically around the Couchbase side and the enum parsing.
On the YAML examples: Yes, update both. The POST example should include emailConfidenceCode: HIGH to show consumers how to use it. The GET response example should also include it. If you leave examples as-is, API documentation tools like Swagger UI will show a response without the new field — confusing for consumers.

Edge case you're missing — fromValue() with unexpected Couchbase data: Your AccountRecord.EmailConfidenceCodeEnum.fromValue() is generated by openapi-generator. The default generated implementation throws IllegalArgumentException for unrecognized strings. If someone bypasses the API and writes a Couchbase document directly with email_confidence_code: "high" (lowercase), your GET will return a 500. Add this to the test scenarios and decide: throw 500, or treat unknown strings as null?

Edge case — concurrent upsert clearing the field: The POST contract says the field is optional. If a client sends a POST without emailConfidenceCode, your toDocument() will set email_confidence_code to null, overwriting a previously stored value. Is that intentional? If account 123 has HIGH and the email verification service sends an update for just the status field without re-sending the confidence code — it gets erased.

Recommendation: Add a sentence to the design doc: "Upsert replaces the full document — clients must re-send emailConfidenceCode on every upsert to preserve it." This is not a bug but it IS a consumer gotcha that should be documented.

📋 BDD/TDD

Use when the behaviour is complex, when product owner sign-off on Gherkin is valuable, or when compliance requires formal test evidence. The ceremony pays off for features with many branches.

🎵 Vibe Coding

Use only for throwaway spikes, local experiments, or solo projects not going to production. In entityapi-db specifically: only for non-API changes where the generated-sources trap cannot be hit.

📝 Markdown-First

Use for any API contract change. The 30-minute design iteration loop prevents hours or days of production incidents, breaking-change cycles, and consumer confusion. Default approach for entityapi-db field additions.

Verdict

💡
Is iterating a Design Markdown with AI before touching the spec a good way to do Spec-Based coding?

Yes — it is arguably the most effective form of Spec-Based Development for API-first projects. The markdown iteration loop uses AI at the highest-value point in the development cycle: before any commitment is made. Every other approach uses AI to generate code. Markdown-First uses AI to refine thinking. The code is almost an afterthought — once the design is solid, the implementation is mechanical.

🏆 Recommended Approach for entityapi-db: Markdown-First Spec-Driven Development

entityapi-db is an OpenAPI-first project. entity-api.yaml is a published contract. Adding a field wrong is expensive to fix — it requires versioning, deprecation, and consumer migration. The Markdown-First loop catches design errors at the cheapest possible moment: before any YAML is committed.

The three AI iteration rounds took ~30 minutes and surfaced: the null-vs-UNKNOWN decision for legacy docs, the upsert-overwrites consumer gotcha, the fromValue() lowercase edge case, and the generated-sources constraint. None of these would have been caught by vibe coding. Only some would have been caught by Spec-Driven V1. All of them are now documented permanently in the design markdown.

The practical workflow for this codebase:
1. Copy design/TEMPLATE.md → fill in what you know
2. Share with AI: "critique this design, find gaps"
3. Revise, repeat until Open Questions is empty
4. Translate to entity-api.yaml (mechanical, no judgment calls)
5. Prompt AI: "implement exactly what the design doc says"
6. Run the acceptance checklist as the definition of done

Combine with BDD/TDD when the business logic is complex or when product-owner sign-off on Gherkin scenarios is required by process. The two approaches are complementary: Markdown-First provides the design rigour; BDD/TDD provides the executable specification. Use both for high-stakes changes.
