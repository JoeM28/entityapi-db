Startup Order & Dependencies

Docker Compose enforces a strict startup order using depends_on + healthcheck conditions. No container starts until its dependencies are confirmed ready.
1
zookeeper starts No dependencies. Starts immediately. Kafka requires it to be up before it can join the cluster.
2
couchbase starts → healthcheck: GET /ui/index.html returns 200 Polls every 10s up to 12 retries. The UI must be accessible before anything tries to connect.
3
kafka starts (depends on: zookeeper) Connects to Zookeeper, registers as broker ID 1, creates internal topics. Advertises two listeners: kafka:29092 for containers and localhost:9092 for host tools.
4
couchbase-setup runs init.sh → exits 0 (depends on: couchbase healthy) Creates the cluster and customer_bucket via couchbase-cli. Idempotent — skips if already exists. The Couchbase source connector cannot start until this bucket exists.
5
kafka-connect starts → healthcheck: GET /connectors returns 200 (depends on: kafka) Loads the Couchbase plugin from /usr/share/confluent-hub-components. Creates three internal Kafka topics for its own state: _connect-configs, _connect-offsets, _connect-status. Takes ~60–90s to fully start.
6
entity-name-api starts → healthcheck: GET /actuator/health (depends on: nothing) Starts independently. entity-api waits for it to be healthy before accepting traffic.
7
kafka-connect-setup runs register-connectors.sh → exits 0 (depends on: kafka-connect healthy + couchbase-setup completed) POSTs couchbase-source.json to http://kafka-connect:8083/connectors. Skips if already registered. Once registered, the connector immediately starts streaming DCP events from Couchbase.
8
entity-api starts (depends on: couchbase-setup completed + entity-name-api healthy) CouchbaseConfig calls bucket.waitUntilReady(10s) — blocks Spring startup until the KV connection is confirmed. Then serves traffic on port 8080.
9
entity-kafka-service starts (depends on: kafka-connect-setup completed) Subscribes to entity.accounts topic as consumer group entity-kafka-email-group. Kafka records its offset so it never processes the same message twice — even after a restart.

POST Request Flow (entity-api)

What happens inside entity-api when you fire POST /entity/account — before Kafka even gets involved.

1
AccountController receives request — @Valid triggers Jakarta Bean Validation Spring automatically validates the AccountRecord object against annotations on the generated model: @NotNull status, @Size(max=20) on legalName, @Pattern on state (2-char uppercase), countryCode (3-char uppercase). Any failure → GlobalExceptionHandler → 400 with errors[] array. Never reaches the service layer.
2
accountService.exists() — Couchbase KV EXISTS 100ms timeout. Returns true/false. No lock, no transaction. Used only to decide the response code: 201 (new document) vs 200 (existing document updated). This check happens before the upsert.
3
accountService.upsert() — manual null + state validation Service-layer guards beyond bean validation: null record → BadDataException → 400. Missing address.state → BadDataException → 400.
4
DobValidator.isValid() — entity-dob-validation Maven library (in-process) Validates MM-DD-YYYY format with a 4-digit year. This is an in-JVM call to the separately versioned library. Invalid → BadDataException → 400.
5
NameApiClient.capitalize() — HTTP POST to entity-name-api:8081 Sends {"legalName":"john smith"}, receives {"legalName":"JOHN SMITH"}. If the call fails for any reason, logs a warning and continues with the original value. Never blocks the upsert.
6
toDocument() — maps AccountRecord → AccountDocument Converts camelCase API model to snake_case Couchbase document. Sets server-side fields: last_update_date (today) and last_update_timestamp (now ISO 8601). Client can never override these.
7
collection.upsert(key, doc) — Couchbase KV UPSERT 500ms timeout. Key = accountNumber as string e.g. "987654321". Inserts if new, replaces if existing. This is a raw Couchbase SDK call — no N1QL, no ORM. At this moment Couchbase emits a DCP mutation event which starts the Kafka pipeline.
8
toRecord() → HTTP 201 or 200 response Maps AccountDocument back to AccountRecord (snake_case → camelCase). Returns the full saved record including server-set fields. Status 201 if the document was new, 200 if it already existed.

Two Model Layers

HTTP boundary (camelCase) Java DTO (generated) Couchbase doc (snake_case) { "legalName": "foo" } ←→ AccountRecord / Name ←→ AccountDocument / NameDoc { "line1": "123 St" } from entity-api.yaml { "legal_name": "FOO" } { "line_1": "123 St" } toDocument(): API model → Couchbase doc (on every write) toRecord(): Couchbase doc → API model (on every read and after write)

Kafka Internals

🦒 Zookeeper — The Coordinator

  • Maintains the list of active Kafka brokers
  • Stores topic metadata and partition assignments
  • Manages leader election for partitions
  • Kafka brokers register themselves here on startup
  • Only used internally — your app never talks to it

📨 Kafka Broker — The Message Store

  • Receives messages from producers (Kafka Connect)
  • Stores them durably in topic partitions on disk
  • Delivers them to consumers (entity-kafka-service)
  • Tracks consumer offsets per group
  • Two listeners: kafka:29092 (internal) and localhost:9092 (host)

📋 Topic: entity.accounts

  • Auto-created by Kafka Connect when first message is published
  • Key: the Couchbase document ID (e.g. "987654321") as a String
  • Value: raw JSON bytes of the full Couchbase document
  • 1 partition (single-node dev setup)
  • Messages are retained for 7 days by default

📍 Consumer Offsets

  • entity-kafka-service uses group ID entity-kafka-email-group
  • After processing each message, it commits the offset back to Kafka
  • If entity-kafka-service restarts, it resumes from where it left off
  • No duplicate emails — each message processed exactly once
  • auto-offset-reset: earliest — on first start, reads from beginning

What is DCP?

Database Change Protocol (DCP)

DCP is Couchbase's internal replication protocol. The Kafka connector opens a persistent DCP stream — think of it like a change-data-capture (CDC) feed. Every time a document is created, updated, or deleted in customer_bucket, Couchbase pushes a DCP event to all DCP subscribers — including the Kafka connector. This happens at the database level, not the application level. Even if you write directly to Couchbase bypassing your API, the event still flows to Kafka.

Connector Configuration

kafka/connectors/couchbase-source.json { "name": "couchbase-source", "config": { "connector.class": "com.couchbase.connect.kafka.CouchbaseSourceConnector", "tasks.max": "1", // Couchbase connection — Docker DNS resolves "couchbase" to the container IP "couchbase.seed.nodes": "couchbase", "couchbase.bucket": "customer_bucket", "couchbase.username": "Administrator", "couchbase.password": "password", // Handler that sends the raw document JSON bytes (no schema wrapper) "couchbase.source.handler": "RawJsonSourceHandler", // All events from customer_bucket go to this Kafka topic "couchbase.topic": "entity.accounts", // key = document ID (String) · value = raw bytes (no schema conversion) "key.converter": "StringConverter", "value.converter": "ByteArrayConverter" } }

How the Connector is Registered

1
kafka-connect-setup container starts (alpine image) Installs curl via apk, strips Windows CRLF from the script (tr -d '\r'), then runs register-connectors.sh
2
Script checks if connector already exists GET http://kafka-connect:8083/connectors/couchbase-source → if 200, skip. If 404, register. This makes it safe to run on every docker compose up.
3
POSTs the connector config to Kafka Connect REST API POST http://kafka-connect:8083/connectors with the JSON body from couchbase-source.json. Connect validates the config, instantiates the connector class, and starts the DCP stream immediately.
4
kafka-connect-setup exits 0 The connector continues running inside the kafka-connect worker. The setup container is done — it's a one-shot job.

entity-kafka-service — Consumer + Email

AccountEventConsumer.java

  • Annotated with @KafkaListener(topics = "entity.accounts")
  • Receives ConsumerRecord<String, byte[]> — key is the document ID, value is raw JSON bytes
  • Converts bytes to String using UTF-8
  • Passes both to EmailNotificationService
  • Kafka auto-commits the offset after the method returns

EmailNotificationService.java

  • Parses the raw JSON bytes and pretty-prints them using Jackson's ObjectMapper
  • Builds subject: "Entity Account Event — Document ID: 987654321"
  • Creates a MimeMessage via Spring's JavaMailSender
  • Sends via Gmail SMTP on port 587 with STARTTLS
  • Catches all exceptions — a failed email never causes a Kafka offset rollback

Gmail SMTP Setup

Why App Password?

Google blocks "less secure" password auth. With 2-Step Verification enabled, you generate a 16-char App Password that only works for SMTP — it can't log into your Google Account. This is how Gmail allows automated email sending safely.

SSL Trust Config

The alpine JRE image has a minimal CA certificate store. Setting mail.smtp.ssl.trust=smtp.gmail.com tells JavaMail to explicitly trust Gmail's SMTP host, bypassing the alpine CA store limitation — without disabling SSL entirely.
