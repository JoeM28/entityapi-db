How the 3 Repos Work Together

Disk layout (all siblings on your machine) ~/ ├── entityapi-db/ ← THIS repo — main Spring Boot service ├── entity-name-api/ ← HTTP microservice (separate repo) └── entity-dob-validation/ ← Maven library (separate repo) sync-sibling-repos.py clones the two sibling repos next to entityapi-db/

◆ entity-dob-validation — Maven Library

A compile-time dependency. Gets baked into the final JAR.

  • Declared in pom.xml like any Maven dep
  • Not on Maven Central — lives in sibling repo
  • You must run mvn install inside it first → publishes to ~/.m2
  • Maven finds it in local cache at compile time
  • Used by: DobValidator.isValid(record.getDob())

◆ entity-name-api — HTTP Microservice

A runtime dependency. A separate running service called over HTTP.

  • Separate Spring Boot app on port 8081
  • Called during every upsert via NameApiClient
  • URL configured via entity.name-api.url in application.yml
  • In Docker: overridden to http://entity-name-api:8081
  • Failure is non-blocking — falls back to original value
Repo Type When Used How Resolved Failure Impact
entity-dob-validation Maven JAR Compile time ~/.m2 local cache Build fails — hard dependency
entity-name-api HTTP Service Runtime (every upsert) Docker DNS / env var URL Falls back to original name — non-blocking
entityapi-db Main Service Always — —

Docker — How Containers Connect

Docker Compose creates a private bridge network called entity-net. All containers join it and resolve each other by service name via Docker's built-in DNS.
┌──────────────────────────────── entity-net (bridge network) ───────────────────────────────┐ │ │ │ [couchbase] [couchbase-setup] [entity-name-api] [entity-api] │ │ port 8091 (UI) one-shot init port 8081 port 8080 │ │ port 11210 (KV) runs init.sh then exits Spring Boot Spring Boot │ │ │ └─────────────────────────────────────────────────────────────────────────────────────────────┘ ↑ only these ports exposed to your host machine (localhost) everything else stays internal — containers talk by name e.g. "couchbase", "entity-name-api"

Startup Order (enforced by depends_on)

1
couchbase starts Healthcheck polls http://localhost:8091/ui/index.html every 10s — waits up to 120s
2
couchbase-setup runs init.sh Creates cluster + customer_bucket via couchbase-cli. Idempotent — safe to re-run. Exits when done.
3
entity-name-api starts Healthcheck polls /actuator/health — waits until healthy
4
entity-api starts Only after step 2 completes AND step 3 is healthy. Connects to Couchbase using Docker DNS name couchbase

⚙ How entity-api Finds Couchbase

Environment variable overrides application.yml:

SPRING_COUCHBASE_CONNECTION_STRING: couchbase # "couchbase" = Docker DNS name of the couchbase container # resolves to the container's internal IP automatically

⚙ How entity-api Finds entity-name-api

Environment variable overrides the localhost default:

ENTITY_NAME_API_URL: http://entity-name-api:8081 # "entity-name-api" = Docker DNS name # without Docker: defaults to http://localhost:8081
CouchbaseConfig.java blocks startup with bucket.waitUntilReady(10s) — the app won't serve traffic until the KV connection is confirmed ready.

Full Request Flow & Validation

POST /entity/account (Upsert)

1
HTTP hits AccountController — @Valid fires Jakarta Bean Validation Checks @NotNull, @Size, @Pattern on the generated AccountRecord model. If any fail → GlobalExceptionHandler returns 400 with field-level errors array. Never reaches the service.
2
accountService.exists() — Couchbase KV EXISTS 100ms timeout. Returns true/false. Used only to decide response status: 201 (new) vs 200 (existing). No lock, no transaction.
3
accountService.upsert() — null + state check Manual validation: null record → 400. Missing address.state → 400. These are service-layer guards beyond what bean validation covers.
4
DobValidator.isValid() — entity-dob-validation library In-process call to the Maven library. Validates MM-DD-YYYY with 4-digit year. Invalid → BadDataException → 400.
5
nameApiClient.capitalize() — HTTP call to entity-name-api POST http://entity-name-api:8081/name/capitalize. Returns capitalized legalName. If it fails for ANY reason → logs warning, uses original value. Non-blocking.
6
collection.upsert() — Couchbase KV UPSERT 500ms timeout. Key = accountNumber as string e.g. "123456789". Inserts or replaces the full document. Sets lastUpdateDate and lastUpdateTimestamp server-side.
7
toRecord() maps document back → AccountRecord → HTTP Response AccountDocument (snake_case Couchbase POJO) → AccountRecord (camelCase API DTO). Returns 201 if new, 200 if existing.

Validation Layers at a Glance

LayerWhat Checks ItExamplesResponse
Bean Validation Jakarta @Valid + annotations on generated model @NotNull status, @Size(max=20) legalName, @Pattern state 400 with errors[] array
State null check Manual check in AccountService Missing address.state 400 BadDataException
DOB format DobValidator from Maven library Must be MM-DD-YYYY with 4-digit year 400 BadDataException
Couchbase error SDK exception Connection failure, timeout 500 (uncaught)

Two Model Layers Explained

HTTP JSON (camelCase) Java API model (generated) Couchbase doc (snake_case) ───────────────────── ────────────────────────── ────────────────────────── { "legalName": "foo" } ←→ AccountRecord / Name ←→ AccountDocument / NameDoc from entity-api.yaml { "legal_name": "FOO" } toDocument() maps API → DB toRecord() maps DB → API Server sets: lastUpdateDate, lastUpdateTimestamp on every write

Scaling — Which Container Gets Hit?

Current setup: One container owns port 8080. Postman always hits that one container — no ambiguity.

✗ Without a Load Balancer

If you try docker compose up --scale entity-api=3 — it fails. Three containers cannot all bind to host port 8080. Docker errors on the 2nd attempt.

✓ With a Load Balancer

Host port is only on the LB. App containers have NO host port binding — only reachable inside entity-net.

  • LB distributes: round-robin by default
  • You can't predict which container a specific request hits
  • nginx, HAProxy, or Traefik are common choices
With a load balancer in front: Postman → localhost:80 │ [nginx / HAProxy] ← only this has a host port / | \ [entity-api-1] [entity-api-2] [entity-api-3] port 8080 port 8080 port 8080 (internal only) (internal only) (internal only)
In Kubernetes / Docker Swarm: The Service object or ingress acts as the built-in load balancer. A request to the service IP gets routed to any healthy pod — random or round-robin depending on the scheduler.

Kubernetes Service vs Istio

◈ Kubernetes Service — Layer 4 (TCP/IP)

Built-in Kubernetes concept. Stable virtual IP + DNS name. Uses kube-proxy (iptables rules) to forward packets.

  • Round-robin by default
  • No HTTP awareness — just TCP
  • No retries, timeouts, or traffic rules
  • "Pick a pod, forward the packet"

◈ Istio — Layer 7 (HTTP/gRPC)

Optional add-on. Injects an Envoy sidecar proxy into every pod. Intercepts all traffic transparently.

  • HTTP-aware routing
  • Retries, timeouts, circuit breaking
  • mTLS between every service pair
  • Distributed tracing built in
With Istio — every pod has a sidecar: entity-api pod entity-name-api pods ┌──────────────────────┐ ┌─────────────────────┐ │ NameApiClient │ │ [name-api-1] │ │ │ │ │ [Envoy sidecar] │ │ [Envoy sidecar] │──── mTLS ────────│ │ └──────────────────────┘ │ [name-api-2] │ │ [Envoy sidecar] │ └─────────────────────┘ ▲ Istio control plane (pushes routing rules to all Envoy sidecars)
FeatureK8s ServiceIstio
Load balancingRound-robin (L4)Round-robin, least-conn, consistent hash (L7)
RetriesNoYes — retry on 503, configurable
TimeoutsNoYes — per route
Circuit breakingNoYes
Traffic splitting (canary)NoYes — weighted routing by %
mTLS between servicesNoYes — automatic
Distributed tracingNoYes — built in with trace IDs
Key relationship: They're complementary — Istio uses the K8s Service for pod discovery, but replaces kube-proxy routing with Envoy sidecars. You still define a Service — Istio reads it to know which pods exist.

How Istio Helps entity-api → entity-name-api at Scale

⚠ Problems Without Istio at Scale

  • If name-api-1 is slow/overloaded, requests still go to it — kube-proxy doesn't know
  • If name-api-1 crashes mid-request, your entity-api gets an error (NameApiClient falls back, but the call still failed)
  • No visibility into which pod is slow or failing
  • No way to do a safe rollout of a new entity-name-api version

① Smarter Load Balancing

Envoy knows response times per pod. Sends new requests to the least-busy pod:

trafficPolicy: loadBalancer: simple: LEAST_CONN # if name-api-2 is slower, # requests go to name-api-1 automatically

② Automatic Retries — Without Changing Code

Retry happens at the Envoy sidecar level. NameApiClient never even sees the error:

retries: attempts: 3 perTryTimeout: 2s retryOn: 5xx,gateway-error,reset

③ Circuit Breaker

If entity-name-api is struggling, Istio stops sending it traffic — prevents cascading failure:

outlierDetection: consecutive5xxErrors: 5 interval: 10s baseEjectionTime: 30s # eject bad pod for 30s, use others

④ Canary Rollout — New Version Safely

Split traffic by weight. Roll forward only when metrics look good:

http: - route: - destination: subset: v1 weight: 90 - destination: subset: v2 weight: 10

⑤ mTLS — Free Encryption

Every call from entity-api to entity-name-api is automatically encrypted and mutually authenticated. Zero certificate management in Java code.

⑥ Distributed Tracing

One trace ID links the full chain across all pods and services:

Postman → entity-api (50ms) └─ entity-name-api (12ms) which pod, how long, status Total: 62ms
Key point: Your NameApiClient code doesn't change at all. Istio operates at the network layer — the Envoy sidecar intercepts traffic transparently. All retry logic, circuit breaking, and load balancing happen outside your application code.

Gloo Mesh vs Istio

Gloo Mesh is NOT a replacement for Istio — it's a management plane built on top of Istio, made by Solo.io. Istio still runs underneath.
Gloo Mesh Management plane — UI, policies, multi-cluster control
▼ manages
Istio Control plane — VirtualService, DestinationRule, mTLS policy
▼ configures
Envoy Sidecars Data plane — actual traffic routing in every pod

The Core Problem Gloo Mesh Solves — Multi-Cluster

With plain Istio at scale — 3 separate islands: Cluster A (us-east) Cluster B (us-west) Cluster C (eu) ├── entity-api ├── entity-api ├── entity-api ├── entity-name-api ├── entity-name-api ├── entity-name-api └── Istio (standalone) └── Istio (standalone) └── Istio (standalone) ↑ separate configs ↑ separate certs ↑ no cross-cluster routing ───────────────────────────────────────────────────────────────────── With Gloo Mesh — single pane of glass: Gloo Mesh Management Plane (single pane of glass) / | \ Cluster A Cluster B Cluster C (Istio) (Istio) (Istio) One policy defined once → pushed to all clusters automatically

How Gloo Mesh Helps Your Scenario Across Clusters

① Cross-Cluster Failover

If all entity-name-api pods in Cluster A go down, Gloo Mesh automatically routes to Cluster B. Plain Istio cannot do this natively.

entity-name-api (Cluster A) ← DOWN │ └─ Gloo Mesh reroutes → entity-name-api (Cluster B)

② Shared mTLS Trust

Plain Istio: each cluster has its own CA — cross-cluster mTLS is painful manual work.

Gloo Mesh: creates a shared root CA across all clusters automatically. entity-api in Cluster A trusts entity-name-api in Cluster B out of the box.

③ One Policy — All Clusters

Plain Istio: write same VirtualService in Cluster A, B, and C. Keep them in sync manually forever.

# Written once, applied everywhere kind: RetryTimeoutPolicy spec: applyToRoutes: - route: labels: app: entity-name-api config: retries: attempts: 3

④ Global Canary Across Clusters

Route 10% of ALL entity-api calls (any cluster) to the new version:

weights: - cluster: cluster-a subset: v2 weight: 5 - cluster: cluster-b subset: v2 weight: 5

Decision Guide — Which Do You Need?

Single cluster? └── Plain Istio is enough └── Too complex to manage raw YAML? └── Gloo Mesh (open source UI layer) Multiple clusters? └── Gloo Mesh — exactly what it's designed for └── Need enterprise support / FIPS / compliance? └── Gloo Mesh Enterprise (paid)
