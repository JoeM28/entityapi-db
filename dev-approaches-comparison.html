Entity API DB — Field Addition Case Study

Adding emailConfidenceCode
Three Different Ways

A ground-level comparison of Traditional BDD/TDD, AI-assisted Vibe Coding, and AI-assisted Spec-Driven Development — using the real entityapi-db codebase as the working example.

+ emailConfidenceCode : string [enum: HIGH | MEDIUM | LOW | UNKNOWN]

What We're Adding

emailConfidenceCode is a new optional field on the AccountRecord indicating how confident the system is in the account's email address. Valid values: HIGH, MEDIUM, LOW, UNKNOWN.

This touches every layer of the entityapi-db stack:

  • entity-api.yaml — OpenAPI spec (source of truth)
  • AccountDocument.java — Couchbase POJO (snake_case)
  • AccountService.java — toDocument() / toRecord() mapping
  • AccountServiceTest.java — unit tests
  • integration-tests.postman_collection.json — integration tests
ℹ️
Because this project uses OpenAPI-first model generation, the generated AccountRecord class is never edited by hand. The field must first appear in entity-api.yaml, then mvn compile regenerates the model automatically.

Approach 1: Traditional BDD / TDD

Behaviour-Driven Development defines what the system should do before writing a line of code. Test-Driven Development enforces Red → Green → Refactor. Both start with intent, not implementation.

BDD / TDD
Core philosophy: You cannot write implementation code unless there is a failing test that demands it. The discipline forces you to think through acceptance criteria, edge cases, and API contracts before touching AccountService.java. In a BDD flavour, the failing tests are written in human-readable Gherkin scenarios that a product owner could review.

1
Define BDD Scenarios (Gherkin)

Before touching any code, the team writes Gherkin feature files describing the expected behaviour of emailConfidenceCode. A product owner, QA engineer, and developer collaborate on these.

features/email_confidence_code.feature 
Feature: Account emailConfidenceCode field

  Background:
    Given the Entity API is running at http://localhost:8080

  Scenario: Store a HIGH confidence code on upsert
    Given a valid account payload with emailConfidenceCode "HIGH"
    When I POST to /entity/account
    Then the response status should be 201
    And the returned emailConfidenceCode should be "HIGH"
    And Couchbase should store email_confidence_code as "HIGH"

  Scenario: Reject invalid confidence code on upsert
    Given a valid account payload with emailConfidenceCode "SUPER_HIGH"
    When I POST to /entity/account
    Then the response status should be 400
    And the error should mention "emailConfidenceCode"

  Scenario: Field is optional — upsert succeeds without it
    Given a valid account payload with no emailConfidenceCode field
    When I POST to /entity/account
    Then the response status should be 201
    And the returned emailConfidenceCode should be absent or null

  Scenario: Retrieve stored confidence code on GET
    Given account 123456789 exists with emailConfidenceCode "MEDIUM"
    When I GET /entity/account/123456789
    Then the returned emailConfidenceCode should be "MEDIUM"
💡
At this point zero code has changed. The team is purely describing intent. A product owner can reject or refine these before any engineer writes implementation code.
2
Write Failing Unit Tests (Red Phase)

Translate the Gherkin scenarios into JUnit 5 unit tests. These are written against AccountService using Mockito — the same pattern as the existing AccountServiceTest.java. They will not compile yet because emailConfidenceCode does not exist on the model.

AccountServiceTest.java — new tests (RED, won't compile) 
@Test
void upsert_storesHighConfidenceCode() {
    AccountRecord record = buildValidRecord();
    // This line fails to compile — field doesn't exist yet
    record.setEmailConfidenceCode(AccountRecord.EmailConfidenceCodeEnum.HIGH);
    when(nameApiClient.capitalize(any())).thenReturn("Test Name");
    when(collection.upsert(any(), any(), any())).thenReturn(null);

    AccountRecord result = service.upsert(record);

    assertNotNull(result.getEmailConfidenceCode());
    assertEquals("HIGH", result.getEmailConfidenceCode().getValue());
}

@Test
void upsert_handlesNullConfidenceCode() {
    AccountRecord record = buildValidRecord();
    // emailConfidenceCode not set — should not throw
    when(nameApiClient.capitalize(any())).thenReturn("Test Name");
    when(collection.upsert(any(), any(), any())).thenReturn(null);

    assertDoesNotThrow(() -> service.upsert(record));
}

@Test
void getById_returnsConfidenceCode_whenStoredInCouchbase() {
    AccountDocument doc = buildDocumentWithConfidenceCode("MEDIUM");
    GetResult mockResult = mock(GetResult.class);
    when(mockResult.contentAs(AccountDocument.class)).thenReturn(doc);
    when(collection.get(eq("123"), any())).thenReturn(mockResult);

    AccountRecord result = service.getById(123L);

    assertEquals("MEDIUM", result.getEmailConfidenceCode().getValue());

Run mvn test now → compile error. That's expected and correct in TDD.

3
Update entity-api.yaml (source of truth)

Only now do we touch the codebase. The OpenAPI spec is updated first because this project is OpenAPI-first: the Java model is auto-generated from it.

entity-api.yaml — AccountRecord schema (diff) 
    AccountRecord:
      type: object
      properties:
        accountNumber:
          type: integer
          format: int64
        status:
          type: string
          enum: [ACTIVE, INACTIVE, SUSPENDED, CLOSED, PENDING, CANCEL]
+       emailConfidenceCode:
+         type: string
+         description: Confidence level in the account's email address
+         enum: [HIGH, MEDIUM, LOW, UNKNOWN]
+         example: HIGH
        dob:
          type: string
          example: "01-15-1990"

Run mvn compile. The generator adds getEmailConfidenceCode() / setEmailConfidenceCode() to AccountRecord and a new EmailConfidenceCodeEnum. Tests now compile but still fail (Red phase) because service mapping is not done.

4
Update AccountDocument (Couchbase POJO)
AccountDocument.java — add snake_case field 
@JsonProperty("email_confidence_code")
private String emailConfidenceCode;

public String getEmailConfidenceCode() { return emailConfidenceCode; }
public void setEmailConfidenceCode(String v) { this.emailConfidenceCode = v; }

Snake_case email_confidence_code matches the Couchbase document convention established by the existing fields (account_number, last_update_date, etc.).

5
Implement Minimum Mapping in AccountService (Green Phase)

Add exactly the mapping needed to make the failing tests pass — no more.

AccountService.java — toDocument() addition 
// In toDocument() — map API enum → Couchbase string
if (r.getEmailConfidenceCode() != null) {
    doc.setEmailConfidenceCode(r.getEmailConfidenceCode().getValue());
}
AccountService.java — toRecord() addition 
// In toRecord() — map Couchbase string → API enum
if (doc.getEmailConfidenceCode() != null) {
    r.setEmailConfidenceCode(
        AccountRecord.EmailConfidenceCodeEnum
            .fromValue(doc.getEmailConfidenceCode())
    );
}

Run mvn test → All tests pass. Green!

6
Update Integration Tests & Postman Collection

Add scenarios to integration-tests.postman_collection.json that mirror the Gherkin scenarios. These run via Newman against the full Docker stack.

Postman test snippet — pm.test assertion 
pm.test("emailConfidenceCode is returned correctly", () => {
    const body = pm.response.json();
    pm.expect(body.emailConfidenceCode).to.equal("HIGH");
});

pm.test("Invalid emailConfidenceCode returns 400", () => {
    pm.expect(pm.response.code).to.equal(400);
    pm.expect(pm.response.json().errors[0].field)
        .to.include("emailConfidenceCode");
});
7
Refactor (if needed) — Cycle Complete

Review the mapping code. Could fromValue() / getValue() enum conversion be centralised? Are there other fields that follow this same pattern? If yes, refactor now while tests protect you. Otherwise: done.

✅ Strengths

  • Every acceptance criterion is captured as a test before code exists
  • Product owner can validate Gherkin before engineers start
  • Impossible to "forget" to handle null — the test forces you to
  • Regression protection: future changes are gated by the test suite
  • CI pipeline catches breakage automatically on every push
  • Clear audit trail: spec → test → implementation

❌ Weaknesses

  • High ceremony for a simple optional field
  • Takes 2–3x longer than just adding the field and testing manually
  • Requires team discipline — pressure shortcuts the Red phase
  • Gherkin step definitions need separate maintenance
  • Value of TDD decreases if team doesn't do regular refactors

Approach 2: AI Vibe Coding

Describe what you want to an AI in natural language, accept the generated code, run it, iterate based on vibes. Fast but highly context-dependent.

Vibe Coding
Core philosophy: Move fast. Describe the intent to an AI, get code back, try it, adjust if it doesn't feel right. There is no formal spec, no tests-first discipline, no pre-agreed acceptance criteria. The engineer is the acceptance test. Works best when the developer deeply understands the codebase and can immediately spot AI hallucinations.

1
The Prompt — Describe Intent to AI

The developer opens Claude Code (or GitHub Copilot, Cursor, etc.) and types:

Developer's vibe prompt (informal, no spec) 
"Hey, I need to add an emailConfidenceCode field to the Account API.
It should be an enum with values HIGH, MEDIUM, LOW, UNKNOWN. It's optional.
It should be stored in Couchbase as email_confidence_code.
Update whatever files you need to — YAML, service, document class, etc."

No Gherkin. No failing test. The developer is trusting the AI to figure out the correct layers to modify and the patterns to follow.

⚠️
The AI has no visibility into the OpenAPI-first constraint unless the developer mentions it. If the AI edits the generated AccountRecord.java directly instead of entity-api.yaml, those changes will be wiped on the next mvn compile. This is a classic vibe coding trap in code-generated projects.
2
AI Generates Code — What It Might Produce

A well-prompted AI on this codebase might produce the correct changes. But without explicit guidance it may also make the following mistakes:

Common AI vibe coding mistake — editing the generated file (WRONG) 
// AI edits target/generated-sources/openapi/.../AccountRecord.java
// This file is REGENERATED by mvn compile — the change will be LOST
// target/generated-sources/openapi/com/company/entityapi/model/AccountRecord.java
public enum EmailConfidenceCodeEnum {
    HIGH("HIGH"), MEDIUM("MEDIUM"), LOW("LOW"), UNKNOWN("UNKNOWN");
    // ^^^^ WRONG — this is in generated-sources, will be overwritten!
}
What the AI should produce (correct path) 
// 1. entity-api.yaml (source of truth)
+     emailConfidenceCode:
+       type: string
+       enum: [HIGH, MEDIUM, LOW, UNKNOWN]

// 2. AccountDocument.java
+   @JsonProperty("email_confidence_code")
+   private String emailConfidenceCode;

// 3. AccountService.java toDocument()
+   if (r.getEmailConfidenceCode() != null)
+       doc.setEmailConfidenceCode(r.getEmailConfidenceCode().getValue());

// 4. AccountService.java toRecord()
+   if (doc.getEmailConfidenceCode() != null)
+       r.setEmailConfidenceCode(AccountRecord.EmailConfidenceCodeEnum
+           .fromValue(doc.getEmailConfidenceCode()));
3
Manual Verification — Fire Up Postman

The developer runs docker compose up -d --build, opens Postman, sends a POST with "emailConfidenceCode": "HIGH", and eyeballs the response. If it looks right, it ships.

Manual Postman test — informal verification 
POST http://localhost:8080/entity/account
{
  "accountNumber": 123456789,
  "docType": "Account",
  "status": "ACTIVE",
  "emailConfidenceCode": "HIGH",  ← new field
  "name": { "legalName": "Test Corp" },

Response: 201 — looks good! Ship it.

But did the dev test:
  ✗ null emailConfidenceCode round-trip?
  ✗ GET after upsert returns the code?
  ✗ Invalid value rejected with 400?
  ✗ Old documents without the field still load?
⚠️
Vibe coding generates no persistent test artifacts. The next developer who changes AccountService.java has no automated safety net for emailConfidenceCode. The regression is only caught if someone manually tests it again.
4
Iteration Loop (The "Vibe")

If the first attempt breaks something (e.g., fromValue() throws on unknown values from old Couchbase documents), the developer re-prompts the AI to fix it. The loop continues until it "feels right."

Follow-up vibe prompt after encountering a bug 
"Getting a NullPointerException when loading old accounts that don't have
the field in Couchbase. Fix it so it handles null gracefully."

— AI fixes, developer tests again, ships

Each iteration is fast but ad-hoc. The fixes don't necessarily become tests, so the same bug can re-emerge later.

⚠️
Critical risk for entityapi-db: This project's AccountRecord is in target/generated-sources/ and is regenerated on every build. An AI doing vibe coding without knowing this will edit the wrong file. There is no failing test to catch it — the bug only surfaces at the next mvn compile and the developer wonders why their changes disappeared.

✅ Strengths

  • Fastest time-to-running-code for simple, well-understood changes
  • Low ceremony — no Gherkin, no test scaffolding upfront
  • AI can identify all affected files at once
  • Great for prototyping or throwaway code
  • Good developer can catch AI mistakes quickly with local testing

❌ Weaknesses

  • No automated regression protection — bugs return silently
  • AI unaware of project-specific constraints (OpenAPI-first, generated sources)
  • Edge cases (null values, old documents) are easily missed
  • Knowledge lives in developer's head, not in tests or spec
  • Code review is harder without a spec to compare against
  • Technical debt accumulates rapidly across many vibe sessions
  • High risk when developer is unfamiliar with the codebase

Approach 3: AI Spec-Driven Development

The OpenAPI spec is updated first as the contract. AI is then used as an accelerator to implement the spec, not a replacement for design. Combines AI speed with spec rigour.

Spec-Driven + AI
Core philosophy: The spec is the single source of truth. A human (or AI-assisted human) updates entity-api.yaml first, creating a precise contract. Then the AI is given the spec diff as context and asked to implement exactly what the spec says — no guessing, no hallucinating field names. This is naturally aligned with how entityapi-db is already structured.

1
Update entity-api.yaml First — Deliberately

The developer (or architect) makes a careful, deliberate update to the OpenAPI spec. This is a design decision, not a code change. Field name, type, enum values, required vs optional, description — all decided here.

entity-api.yaml — full, precise spec change 
    AccountRecord:
      type: object
      properties:
        accountNumber:
          type: integer
          format: int64
        status:
          type: string
          enum: [ACTIVE, INACTIVE, SUSPENDED, CLOSED, PENDING, CANCEL]
+       emailConfidenceCode:
+         type: string
+         description: |
+           Indicates how confident the system is that the account's email
+           address is valid and reachable.
+           HIGH   — confirmed deliverable
+           MEDIUM — likely valid, not verified
+           LOW    — suspected invalid or unreachable
+           UNKNOWN — no email verification data available
+         enum: [HIGH, MEDIUM, LOW, UNKNOWN]
+         example: HIGH
💡
The spec diff is a reviewable artifact. A product owner, API consumer team, or tech lead can review and approve the spec change in a PR before any implementation work begins.
2
Provide Spec Diff to AI as Authoritative Context

Unlike vibe coding where the AI guesses at intent, here the AI is given the spec diff and the project conventions, and asked to implement exactly what the spec describes.

Precise AI prompt (spec-driven, not vibe) 
"I've just updated entity-api.yaml to add emailConfidenceCode (enum:
HIGH | MEDIUM | LOW | UNKNOWN, optional). This project is OpenAPI-first:
AccountRecord is generated from the YAML by openapi-generator-maven-plugin
into target/generated-sources/ — do not edit it.

Please implement exactly what the spec says across:
  1. AccountDocument.java — add email_confidence_code (snake_case, @JsonProperty)
  2. AccountService.toDocument() — map EmailConfidenceCodeEnum → String
  3. AccountService.toRecord()   — map String → EmailConfidenceCodeEnum
  4. AccountServiceTest.java     — add unit tests for all four enum values
                                   and for null (field absent)

Follow the exact patterns of existing fields: status uses getValue()/fromValue(),
use the same pattern. Do not add validation — the enum in the YAML handles it."

The AI is now working from a contract, not a description. It cannot hallucinate field names or types because the spec is explicit.

3
AI Implements All Layers (Following Spec)

The AI produces all four file changes in one pass, guided by the spec:

AccountDocument.java — AI output, spec-guided

private String emailConfidenceCode;

public String getEmailConfidenceCode() { return emailConfidenceCode; }
public void setEmailConfidenceCode(String v) { this.emailConfidenceCode = v; }

}

// toRecord() — mirrors status field pattern exactly

4
AI Generates Tests Derived from the Spec

Because the spec is explicit about enum values, the AI generates a complete test matrix from the spec — not just a happy path.

AccountServiceTest.java — AI-generated spec-derived tests 
@ParameterizedTest
@EnumSource(AccountRecord.EmailConfidenceCodeEnum.class)
void upsert_storesAllValidConfidenceCodes(
        AccountRecord.EmailConfidenceCodeEnum code) {
    // Spec defines 4 valid values — test ALL of them, not just HIGH
    AccountRecord record = buildValidRecord();
    record.setEmailConfidenceCode(code);
    when(nameApiClient.capitalize(any())).thenReturn("Test Corp");

    AccountRecord result = service.upsert(record);

    assertEquals(code, result.getEmailConfidenceCode());
}

@Test
void upsert_acceptsAbsent_emailConfidenceCode() {
    // Spec says field is optional — null must be handled
    AccountRecord record = buildValidRecord();
    // emailConfidenceCode intentionally not set
    when(nameApiClient.capitalize(any())).thenReturn("Test Corp");

    assertDoesNotThrow(() -> service.upsert(record));
}

@Test
void toRecord_mapsConfidenceCode_fromCouchbaseDocument() {
    // Spec: GET /entity/account/{id} must return emailConfidenceCode
    AccountDocument doc = buildDocumentWithConfidenceCode("LOW");
    GetResult mockResult = mock(GetResult.class);
    when(mockResult.contentAs(AccountDocument.class)).thenReturn(doc);
    when(collection.get(eq("999"), any())).thenReturn(mockResult);

    AccountRecord result = service.getById(999L);

    assertEquals(AccountRecord.EmailConfidenceCodeEnum.LOW,
                 result.getEmailConfidenceCode());
}

@Test
void toRecord_handlesLegacyDocument_withoutConfidenceCode() {
    // Spec: field is optional — old Couchbase docs won't have it
    AccountDocument doc = buildDocumentWithConfidenceCode(null);
    GetResult mockResult = mock(GetResult.class);
    when(mockResult.contentAs(AccountDocument.class)).thenReturn(doc);
    when(collection.get(eq("888"), any())).thenReturn(mockResult);

    AccountRecord result = service.getById(888L);

    assertNull(result.getEmailConfidenceCode());
}

The AI derives the legacy document test from the spec's optional designation — something a vibe coder often misses until production breaks.

✅ Strengths

  • Spec is the authoritative contract — AI can't hallucinate field names
  • Aligns with entityapi-db's existing OpenAPI-first architecture
  • AI generates tests derived from the spec (all enum values, null handling)
  • Spec change is reviewable before implementation begins
  • Fast: AI handles all implementation layers in one pass
  • Regression protection via generated tests + CI
  • New team members can onboard via the spec, not tribal knowledge

❌ Weaknesses

  • Requires developer to understand the spec format (YAML/OpenAPI)
  • More upfront effort than vibe coding — spec must be done right
  • AI still needs good context about project conventions (prompt quality matters)
  • Over-reliance on AI means developers may not deeply understand the code

When to Use Which Approach

The right approach depends on the team, the change, and the risk.

Situation 📋 BDD/TDD 🎶 Vibe Coding 📄 Spec-Driven AI
Critical production field, financial data Best choice Avoid Good choice
New field on existing schema (like this example) Good Risky Best choice
Rapid prototype or internal tool Overkill Best choice Fine
External API consumed by other teams Good Avoid Best choice
Complex business logic with many edge cases Best choice Avoid Good with AI-generated tests
Solo developer who knows codebase deeply Fine Good Best choice
New developer onboarding Best choice High risk Good — spec guides them
Regulatory or compliance requirement Required Unacceptable Good if spec is auditable

📋 Use BDD/TDD When…

  • The change involves complex business rules
  • Multiple scenarios need product owner sign-off
  • Regulatory compliance requires test evidence
  • The team values long-term test suite quality
  • You're defining new behavior, not modifying existing

🎶 Use Vibe Coding When…

  • It's a throwaway prototype or spike
  • You deeply know the codebase and its traps
  • Speed is the only constraint
  • The output will be rewritten properly later
  • There's no downstream consumer of the API

📄 Use AI Spec-Driven When…

  • The project is already OpenAPI-first (like this one)
  • Other teams consume the API and need a stable contract
  • You want AI speed with spec discipline
  • The change is well-scoped with clear acceptance criteria
  • You want test coverage without full TDD ceremony

Verdict for entityapi-db

🏆 Recommended Approach: AI Spec-Driven Development

entityapi-db is already an OpenAPI-first project. The entity-api.yaml spec is the declared source of truth. AccountRecord is a generated artifact in target/ — editing it directly (a classic vibe coding failure) is impossible to do safely. The project has an established pattern: YAML → generated model → document class → service mapping.

AI Spec-Driven Development is a natural extension of this design. Update the YAML first (one file, reviewable diff), then give the AI the spec change and project conventions as context. The AI implements all layers correctly because the field name, type, and constraints are unambiguous in the spec. Test cases are derived from the spec's enum values and optional designation, meaning the null/legacy-document edge case is caught before production.

BDD/TDD is the right evolution when the team scales, when Gherkin scenarios need product owner approval, or when compliance requires formal test evidence. For a small team moving fast, AI Spec-Driven gives you 80% of the protection at 30% of the ceremony.

Vibe coding is appropriate only for local spikes or throwaway experiments in this codebase. The OpenAPI-first architecture and generated sources create specific traps that an AI without explicit context will fall into.
