Request Flow — What Actually Happens

Key insight: In both approaches, the final Couchbase operation is identical — collection.get() or collection.upsert(). The difference is whether you write that line or Spring Data does it invisibly inside a generated proxy class.

With Spring Data, a hidden proxy class sits between your service and the Couchbase SDK. You cannot see it, step into it in the debugger, or configure it per-operation.

AccountRepository.java

Spring Data interface vs no repository needed with raw SDK

AccountRepository.java

src/main/java/…/repository/
Spring Data (current) — 14 lines 
@Repository
public interface AccountRepository
    extends CouchbaseRepository<AccountDocument, String> {

  // No methods written here.
  // Spring Data generates a proxy class
  // at runtime that implements:
  //
  //   findById(String id)
  //   existsById(String id)
  //   save(AccountDocument doc)
  //
  // You have NO visibility into what
  // those implementations do, what
  // SDK options they use, or how
  // errors are handled.
}
Raw SDK — repository file deleted entirely 
// AccountRepository.java → DELETED
//
// With raw SDK you inject the
// Couchbase Collection directly
// into AccountService.
//
// No proxy. No magic. No hidden layer.
// Every DB call is a line of code
// you wrote yourself.
//
// The Collection bean is
// auto-configured by Spring Boot
// from application.yml — you just
// inject it where you need it.
Why the interface feels convenient but isn't: The 14-line interface looks clean, but Spring Data generates hundreds of lines of proxy code behind it. You cannot read that code, set breakpoints in it, or configure it. When something goes wrong, your stack trace goes into Spring Data internals — not your code.

AccountService.java

Full side-by-side — every DB call made explicit

READ — getById()

GET /entity/account/{accountId}
Spring Data (current) 
public AccountRecord getById(Long accountId) {

    return repository
        .findById(String.valueOf(accountId))
        .map(this::toRecord)
        .orElseThrow(() ->
            new AccountNotFoundException(accountId));

    // What actually runs:
    //   collection.get(key)    ← hidden
    //
    // What you CANNOT control:
    //   - timeout (uses cluster default)
    //   - replica reads (always primary)
    //   - return expiry alongside doc
}
Raw SDK (explicit) 
public AccountRecord getById(Long accountId) {
    try {
        GetResult result = collection.get(
            String.valueOf(accountId),
            GetOptions.getOptions()
                .timeout(Duration.ofMillis(200))
                .withExpiry(true)
        );
        return toRecord(
            result.contentAs(AccountDocument.class));

    } catch (DocumentNotFoundException e) {
        throw new AccountNotFoundException(accountId);
    }
    // Every option is visible and tunable.
    // Timeout, replica reads, expiry
    // — all in your code, not hidden.
}

EXISTS check — before upsert to decide 200 vs 201

POST /entity/account
Spring Data (current) 
public boolean exists(Long accountNumber) {

    return repository
        .existsById(String.valueOf(accountNumber));

    // Calls collection.exists() under the hood.
    // Timeout, options — all hidden.
}
Raw SDK (explicit) 
public boolean exists(Long accountNumber) {

    return collection.exists(
        String.valueOf(accountNumber),
        ExistsOptions.existsOptions()
            .timeout(Duration.ofMillis(100))
    ).exists();
    // Explicit. Tunable. Readable.
}

WRITE — upsert()

POST /entity/account
Spring Data (current) 
public AccountRecord upsert(AccountRecord record) {

    AccountDocument saved =
        repository.save(toDocument(record));

    return toRecord(saved);

    // What actually runs:
    //   collection.upsert(key, doc)  ← hidden
    //
    // What you CANNOT set:
    //   - durability level
    //   - per-document TTL (expiry)
    //   - CAS for optimistic locking
    //   - timeout
}
Raw SDK (explicit) 
public AccountRecord upsert(AccountRecord record) {

    AccountDocument doc = toDocument(record);

    collection.upsert(
        doc.getId(),
        doc,
        UpsertOptions.upsertOptions()
            .durability(DurabilityLevel.MAJORITY)
            .expiry(Duration.ofHours(24))
            .timeout(Duration.ofMillis(500))
    );
    return toRecord(doc);
    // Durability, TTL, timeout — all yours.
}

4 Reasons the Abstraction is a Liability

Each one becomes a real problem the moment you need to tune or scale
1

KV Options — Timeout, Durability, TTL are invisible

Spring Data uses cluster-level defaults for every operation. You cannot override them per call.
What Spring Data does
// Spring Data calls this internally —
// you have no way to change these:

collection.upsert(key, content);

// Timeout    → cluster default (2.5s)
//              cannot change per-call
// Durability → NONE — write acknowledged
//              before reaching replicas
//              (data loss risk on node fail)
// Expiry/TTL → 0 — document never expires
//              even if you want cache-style
//              auto-deletion
What raw SDK lets you do
// You control every option per call:

collection.upsert(key, content,
    UpsertOptions.upsertOptions()
        .timeout(Duration.ofMillis(500))
        .durability(
            DurabilityLevel.MAJORITY)
        .expiry(Duration.ofHours(24))
);
// Read with fast timeout for
// latency-sensitive endpoints:
collection.get(key,
    GetOptions.getOptions()
        .timeout(Duration.ofMillis(100)));
2

CAS — Concurrent Updates Always Silently Overwrite Each Other

Spring Data's save() calls upsert() which ignores the CAS (Compare-And-Swap) token. Last writer always wins, with no error.
Spring Data — last write wins silently
// Thread A reads accountNumber=123
// Thread B reads accountNumber=123
// Thread A updates status → SUSPENDED
// Thread B updates status → ACTIVE
//
// Both call repository.save() →
// Both call collection.upsert() →
// Thread B wins. Thread A's write
// is silently discarded.
// No error. No warning. Data lost.

AccountDocument doc = toDocument(record);
repository.save(doc);
// CAS token is never used.
Raw SDK — optimistic locking with CAS
// Read the document AND its CAS token:
GetResult result = collection.get(key);
long cas = result.cas();

AccountDocument doc = toDocument(record);

try {
    collection.replace(key, doc,
        ReplaceOptions.replaceOptions()
            .cas(cas));
} catch (CasMismatchException e) {
    // Another thread modified the doc.
    // Retry or return 409 Conflict —
    // your choice. Data is never lost.
    throw new ConcurrentModificationException();
}
3

N1QL Query Tuning — You Cannot See or Control Generated Queries

The moment you add a findByStatus() method, Spring Data generates a N1QL query you never wrote and cannot tune.
Spring Data — query you did not write
// You write this in AccountRepository:
List<AccountDocument> findByStatus(
    String status);

// Spring Data generates and runs this
// N1QL — you never see it:
SELECT META(d).id, d.*
FROM   customer_bucket d
WHERE  d.status = "ACTIVE"
AND    d._class = "com.company...AccountDocument"

// Problems:
//  - No index hint
//  - _class filter is hidden overhead
//  - No LIMIT or pagination
//  - No scan consistency control
//  - Full bucket scan if index missing
Raw SDK — query you own completely
// You write the exact N1QL:
String sql =
  "SELECT META(d).id, d.* " +
  "FROM customer_bucket d " +
  "WHERE d.doc_type = $type " +
  "AND d.status = $status " +
  "ORDER BY d.last_update_timestamp DESC " +
  "LIMIT $limit OFFSET $offset";

QueryResult r = cluster.query(sql,
    QueryOptions.queryOptions()
        .parameters(JsonObject.create()
            .put("type",   "Account")
            .put("status", status)
            .put("limit",  pageSize)
            .put("offset", page * pageSize))
        .scanConsistency(
            REQUEST_PLUS)  // strong read
        .readonly(true));  // read hint
4

Sub-Document Operations — Full Document Replace vs Surgical Field Update

Spring Data always replaces the entire document. Couchbase's sub-doc API lets you update one field — huge difference at scale.
Spring Data — whole document on the wire
// You want to update only status.
// Spring Data has no sub-doc support.
// It always does a full replace:

// 1. Read entire document   (e.g. 5 KB)
AccountDocument doc =
    repository.findById(key).get();

// 2. Change one field in Java memory
doc.setStatus("SUSPENDED");
doc.setLastUpdateTimestamp(
    Instant.now().toString());

// 3. Write entire document  (5 KB again)
//    Even though only 2 fields changed.
repository.save(doc);

// Network: 5 KB read + 5 KB write = 10 KB
// for a 20-byte logical change.
Raw SDK — only the changed fields on the wire
// Sub-document mutateIn: surgical update.
// Only the changed fields travel
// over the network:

collection.mutateIn(
    String.valueOf(accountNumber),
    List.of(
        MutateInSpec.upsert(
            "status",
            "SUSPENDED"),
        MutateInSpec.upsert(
            "last_update_timestamp",
            Instant.now().toString())
    )
);

// Network: ~30 bytes.
// No read needed. Atomic on the server.
// Document never leaves Couchbase.

Verdict

When to use each approach

Use Spring Data when…

  • → Prototype or internal tool with low traffic
  • → All operations are simple key lookups (no queries)
  • → You never need per-document TTL or durability
  • → Concurrent writes to the same document are impossible
  • → You want the fastest possible path to a working API

Use Raw SDK when…

  • → Production system with SLA requirements
  • → Multiple threads can update the same document
  • → You need N1QL queries you can read, EXPLAIN, and tune
  • → Per-document TTL or durability matters
  • → You will need sub-document updates for performance

For this Entity API — use Raw SDK

  • Account records can be updated concurrently (two requests updating the same accountNumber at once) — CAS is needed to prevent silent data loss.
  • Status updates change one field on a potentially large document — mutateIn() is 10× more efficient than a full replace.
  • The GET endpoint is on a latency-sensitive path — a per-call 200ms timeout prevents a slow Couchbase node from hanging the whole request.
  • Future requirement to query by status will need a tuned N1QL query with the right index and scan consistency — writing it yourself is non-negotiable.
  • The Spring Data version already compiles and works. Rewriting the service layer to use raw SDK means changing ~50 lines in AccountService.java and deleting AccountRepository.java. The controller, document class, and generated models stay exactly as-is.
