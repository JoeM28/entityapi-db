Reusable Schemas — components/schemas
Defined once, referenced everywhere via $ref: '#/components/schemas/Name' — change in one place, all endpoints update automatically
Why components/schemas? Without a shared component, the Name and Address definitions would be copy-pasted into both AccountRecord and CustomerRecord. If you later added a line2 field to Address you would have to find and change it in every place. With $ref you define it once and reference it everywhere.

  type: object

Reusable Responses — components/responses
Pre-built HTTP response definitions shared across endpoints via $ref — ensures consistent error format everywhere
Why components/responses? Every endpoint that can fail must define its error response. Without reusable responses you would copy-paste the same 400/404/500 definition into each of the 5 endpoints — 15+ repetitions. With $ref: '#/components/responses/BadRequest' you define it once. Change the error schema once and all 5 endpoints inherit it automatically.
400 BadRequest The request body or query parameters failed validation

Triggered when incoming data violates any schema rule — wrong pattern, value over maxLength, invalid enum value, missing required field, or wrong data type. The errors[] array tells the caller exactly which fields failed and why, so they can fix the request without guessing.

Common triggers
state not ^[A-Z]{2}$ countryCode not ^[A-Z]{3}$ legalName over 20 chars dob wrong format status unknown value missing required field
Used by endpoints
GET /entity/account POST /entity/account GET /entity/account/{id} GET /entity/customer POST /entity/customer GET /entity/customer/{id}
404 NotFound No record exists with the provided ID

Only used by the two GET-by-ID endpoints. Returned when the path parameter ID does not match any document in the datastore. Not used on POST (upsert creates new if not found) or GET-all (returns empty array, not 404).

Used by endpoints
GET /entity/account/{accountId} GET /entity/customer/{customerId}
Example 404 body
{
  "status":    404,
  "error":     "Not Found",
  "message":   "No Account found with accountNumber 999999999",
  "timestamp": "2026-03-01T10:30:00Z",
  "path":      "/entity/account/999999999"
}
500 InternalServerError Unexpected server-side failure

Returned when something fails that is outside the caller's control — database unreachable, unhandled exception, infrastructure fault. Message is intentionally generic to avoid leaking internal details. Every endpoint includes this because any operation can fail at the infrastructure level.

Used by endpoints
GET /entity/account POST /entity/account GET /entity/account/{id} GET /entity/customer POST /entity/customer GET /entity/customer/{id}

Key Design Principles
Architectural decisions made in this spec and the reasoning behind each one
01
readOnly on Server-Set Timestamps readOnly: true

lastUpdateDate and lastUpdateTimestamp are marked readOnly: true in both AccountRecord and CustomerRecord. This means they appear in response bodies but Swagger UI strips them from the request body editor and code generators exclude them from request models. The server always sets these fields on every POST — the caller never sends them. If a caller does include them the server ignores them. This prevents stale or fabricated timestamps from being injected into the datastore.

02
docType Locked Per Endpoint enum: [Account]

On AccountRecord the docType enum contains only "Account". On CustomerRecord it contains only "Customer". This means if a caller POSTs to /entity/account with docType: "Customer" in the body, schema validation fails immediately with a 400 — no request reaches the service layer. The URL and the document type are always guaranteed to be in agreement. This eliminates an entire class of data corruption bug where Account documents could be written to the Customer path or vice versa.

03
GET All + POST Share the Same Path multi-method path

Both GET /entity/account and POST /entity/account live under the same path block in the YAML. This is standard REST design — the URL represents the resource collection, and different HTTP methods express different operations on that collection. GET retrieves the collection. POST adds to or updates within the collection. Having them on the same path is not a mistake — it is intentional REST convention.

04
Pagination Envelope on GET All wrapper object

The GET all endpoints return { totalRecords, page, size, records: [...] } rather than a raw JSON array [...]. A raw array gives the caller no way to know how many total records exist — they cannot calculate how many pages there are or whether they have reached the end. The wrapper object solves this. totalRecords: 150 with size: 20 tells the caller there are 8 pages total even though only 20 records were returned in this response.

05
POST as Upsert — 200 vs 201 Signal 200 update · 201 insert

Rather than separate POST (create) and PUT (update) endpoints, POST handles both via upsert logic. The response code distinguishes the outcome — 200 means the record already existed and was updated, 201 means a new record was created. This lets the caller know whether they performed an insert or an update without needing to first do a GET to check existence. It also reduces the API surface from 7 endpoints down to 5.

06
Name + Address Extracted as Shared Schemas $ref reuse

The Name and Address objects are defined once in components/schemas and referenced in both AccountRecord and CustomerRecord via $ref. If a new field like line2 is added to Address it is changed in one place and both record types automatically pick it up. Without this the same definition would be duplicated — two places to keep in sync, two places to forget to update, two places where field-level rules like the ^[A-Z]{2}$ state pattern could drift out of sync.

07
ErrorResponse Schema Centralised consistent error contract

All three reusable responses — BadRequest, NotFound, InternalServerError — use the same ErrorResponse schema. Every error the API returns has the same shape: status, error, message, timestamp, path, and errors[]. This gives callers a consistent contract — they can write one error handler that works for every endpoint and every error type, rather than special-casing each one. The errors[] array carries field-level detail on 400s so the caller knows exactly which fields to fix.

08
404 Only on GET-by-ID, Not on GET-All empty array not 404

NotFound is only referenced by GET /entity/account/{accountId} and GET /entity/customer/{customerId}. The GET-all endpoints do not include 404 — if no records match the filter they return 200 with an empty records: [] array. Returning 404 for an empty collection would be wrong — the collection itself exists, it just has no matching members. 404 means the specific resource identified by the path does not exist, which only applies to the single-ID lookup.

entity-api.yaml · OpenAPI 3.0.3 
openapi: 3.0.3

info:
  title: Entity API
  description: |
    REST API for managing Customer Account entities.

    URL Structure:
    - Account resource:    /entity/account
    - Customer resource:   /entity/customer

    Operations:
    - GET  /entity/account               — Retrieve all Account records
    - GET  /entity/account/{accountId}   — Retrieve one Account by ID
    - POST /entity/account               — Insert or Update an Account
    - GET  /entity/customer              — Retrieve all Customer records
    - GET  /entity/customer/{customerId} — Retrieve one Customer by ID
    - POST /entity/customer              — Insert or Update a Customer
  version: 1.0.0
  contact:
    name: API Support
    email: api-support@company.com

servers:
  - url: https://api.company.com/v1
    description: Production server
  - url: https://api-dev.company.com/v1
    description: Development server

tags:
  - name: Account
    description: Account sub-resource operations
  - name: Customer
    description: Customer sub-resource operations

# ─────────────────────────────────────────────────────────────────
# PATHS
# ─────────────────────────────────────────────────────────────────
paths:

  # GET /entity/account + POST /entity/account
  /entity/account:
    get:
      tags: [Account]
      summary: Retrieve all Account records
      operationId: getAllAccounts
      parameters:
        - name: status
          in: query
          required: false
          schema:
            type: string
            enum: [ACTIVE, INACTIVE, SUSPENDED, CLOSED, PENDING, CANCEL]
        - name: page
          in: query
          schema: { type: integer, default: 0, minimum: 0 }
        - name: size
          in: query
          schema: { type: integer, default: 20, minimum: 1, maximum: 100 }
      responses:
        '200':
          description: Paginated list of Account records
          content:
            application/json:
              schema:
                type: object
                properties:
                  totalRecords: { type: integer }
                  page:         { type: integer }
                  size:         { type: integer }
                  records:
                    type: array
                    items:
                      $ref: '#/components/schemas/AccountRecord'
        '400': { $ref: '#/components/responses/BadRequest' }
        '500': { $ref: '#/components/responses/InternalServerError' }

    post:
      tags: [Account]
      summary: Insert or Update an Account (Upsert)
      operationId: upsertAccount
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/AccountRecord'
      responses:
        '200':
          description: Account updated successfully
          content:
            application/json:
              schema: { $ref: '#/components/schemas/AccountRecord' }
        '201':
          description: Account created successfully
          content:
            application/json:
              schema: { $ref: '#/components/schemas/AccountRecord' }
        '400': { $ref: '#/components/responses/BadRequest' }
        '500': { $ref: '#/components/responses/InternalServerError' }

  # GET /entity/account/{accountId}
  /entity/account/{accountId}:
    get:
      tags: [Account]
      summary: Retrieve one Account by ID
      operationId: getAccountById
      parameters:
        - name: accountId
          in: path
          required: true
          schema: { type: integer, format: int64 }
          example: 123456789
      responses:
        '200':
          description: Account record found
          content:
            application/json:
              schema: { $ref: '#/components/schemas/AccountRecord' }
        '400': { $ref: '#/components/responses/BadRequest' }
        '404': { $ref: '#/components/responses/NotFound' }
        '500': { $ref: '#/components/responses/InternalServerError' }

        - name: size
          in: query
          schema: { type: integer, default: 20, maximum: 100 }
      responses:
        '200':
          description: Paginated list of Customer records
          content:
            application/json:

                type: object

                  totalRecords: { type: integer }

                    type: array
                    items:
                      $ref: '#/components/schemas/CustomerRecord'

          schema: { $ref: '#/components/schemas/ErrorResponse' }
