Regular Expression (Regex) Guide

Pattern validation used across OpenAPI fields
What is a Regex? A Regular Expression is a sequence of characters that defines a search pattern. In OpenAPI schemas the pattern keyword uses regex to validate that a field value matches an expected format before it is accepted.

Regex Building Blocks

Character Classes · Quantifiers · Anchors
Core Regex Symbols
ANCHORS — Control where the pattern starts and ends
  ^          Start of string — nothing can appear before this point
  $          End of string — nothing can appear after this point
  ^ABC$      Entire string must be exactly "ABC" — no extra chars allowed

CHARACTER CLASSES — Define what characters are allowed
  [A-Z]     Any uppercase letter A through Z
  [a-z]     Any lowercase letter a through z
  [0-9]     Any digit 0 through 9
  [A-Za-z]  Any letter upper or lower case
  [^abc]    Any character EXCEPT a, b, or c (^ inside [] means NOT)

QUANTIFIERS — Define how many times a character or group repeats
  {3}       Exactly 3 times
  {2,5}     Between 2 and 5 times (inclusive)
  {1,}      1 or more times (same as +)
  +          1 or more times
  *          0 or more times
  ?          0 or 1 time (optional)

GROUPS AND ALTERNATION
  (ABC)     Group — treat ABC as a single unit
  A|B       OR — matches A or B
  (A|B)     Group OR — matches A or B as a unit
  (AB|CD)   Matches either "AB" or "CD"

LITERALS
  -          Literal hyphen (when outside [ ] or at start/end inside)
  .          Any single character (except newline) — use \. for literal dot
  \d         Any digit (same as [0-9])
  \w         Any word character (letters, digits, underscore)

Every Pattern Used in This Schema — Explained

Field by Field Breakdown
Country Code — ^[A-Z]{3}$
  ^          Start of string
  [A-Z]     Any uppercase letter A through Z — no lowercase, no numbers, no symbols
  {3}       Exactly 3 of the above characters
  $          End of string

  Valid:    USA   GBR   CAN   MEX
  Invalid:  us    (lowercase)
            US    (only 2 chars)
            US1   (contains number)
            U.S.  (contains dots)
State Code — ^[A-Z]{2}$
  ^          Start of string
  [A-Z]     Any uppercase letter
  {2}       Exactly 2 characters
  $          End of string

  Valid:    TX   NY   CA   FL
  Invalid:  tx   (lowercase)   Texas  (too long)   T  (too short)
Date of Birth — ^(0[1-9]|1[0-2])-(0[1-9]|[12][0-9]|3[01])-[0-9]{4}$
  The pattern has 3 sections separated by literal hyphens: MM - DD - CCYY

  ── MONTH SECTION: (0[1-9]|1[0-2]) ──────────────────────────────────────
  (           Open group
    0[1-9]    First digit is 0, second digit is 1-9  → covers months 01 to 09
    |         OR
    1[0-2]    First digit is 1, second digit is 0,1,2 → covers months 10, 11, 12
  )           Close group
  Result:  Accepts 01 02 03 04 05 06 07 08 09 10 11 12
  Rejects: 00 (no month zero)  13 (no 13th month)

  -           Literal hyphen separator

  ── DAY SECTION: (0[1-9]|[12][0-9]|3[01]) ───────────────────────────────
  (
    0[1-9]    Days 01 through 09
    |
    [12][0-9] First digit 1 or 2, second digit 0-9 → covers days 10 through 29
    |
    3[01]     First digit 3, second digit 0 or 1 → covers days 30 and 31
  )
  Result:  Accepts 01-09, 10-29, 30, 31
  Rejects: 00   32   99

  -           Literal hyphen separator

  ── YEAR SECTION: [0-9]{4} ───────────────────────────────────────────────
  [0-9]     Any digit 0-9
  {4}       Exactly 4 digits
  Result:  Accepts 1990  2000  2026  0001
  Rejects: 90  (too short)  20260  (too long)

  Full Valid Examples:   01-15-1990   12-31-2000   07-04-1776
  Full Invalid Examples: 1-5-1990 (missing leading zeros)
                         2000-01-15 (wrong order CCYY-MM-DD)
                         01/15/1990 (slashes not hyphens)
Status Enum — Why Not Use Regex?
  Status uses enum instead of regex because the allowed values are fixed and known upfront:
  enum: ["ACTIVE", "INACTIVE", "SUSPENDED", "CLOSED", "PENDING", "CANCEL"]

  Enum is cleaner than regex for fixed value lists because:
  → Regex equivalent would be: ^(ACTIVE|INACTIVE|SUSPENDED|CLOSED|PENDING|CANCEL)$
  → Enum is more readable, self-documenting, and generates better validation errors
  → Enum also auto-generates dropdown choices in tools like Swagger UI

OpenAPI JSON Schema

camelCase · REST API Contract · Validation Rules
Convention: OpenAPI schemas use camelCase — the de facto standard for REST APIs and JSON originating from JavaScript conventions. This is your API contract that validates all incoming and outgoing data.

Understanding type: object and properties

OpenAPI Fundamentals
What is type?
type tells OpenAPI what kind of value this field holds.
It is the data type declaration for that field.

OpenAPI has 6 built-in types:

  string     Any text value
             "ACTIVE", "Acme Corp", "TX"

  integer    Whole numbers — no decimals
             123456789, 42, 0

  number     Any number including decimals
             3.14, 100.50

  boolean    True or false only
             true, false

  array      A list of values
             ["ACTIVE", "PENDING", "CLOSED"]

  object     A group of named fields (key-value pairs)
             { "legalName": "Acme", "dbaName": "Acme Corp" }

In your schema:
  "docType"       → type: string   (text value)
  "accountNumber" → type: integer  (whole number)
  "customerId"    → type: integer  (whole number)
  "name"          → type: object   (contains sub-fields)
  "address"       → type: object   (contains sub-fields)
  "status"        → type: string   (text + enum restriction)
  "dob"           → type: string   (text with pattern)
What is type: object?
type: object means the field is not a simple value
like a string or number. Instead it is a CONTAINER
that holds a group of named sub-fields inside it.

Think of it like a folder that contains files inside.
The object itself has no value — it is just a wrapper
that organizes related fields together.

In your schema there are 3 objects:

  Root object — the entire CustomerAccount record
    Contains: docType, accountNumber, name,
              address, dob, status, customerId,
              lastUpdateDate, lastUpdateTimestamp

  name object — groups name-related fields
    Contains: legalName, dbaName

  address object — groups address-related fields
    Contains: line1, city, state, countryCode

JSON representation of an object looks like:
{                         ← opening brace = object start
  "legalName": "Acme",   ← sub-field inside the object
  "dbaName":   "Acme Corp" ← another sub-field
}                         ← closing brace = object end

type: object in JSON API payload:
{
  "name": {              ← name is an object (has braces {})
    "legalName": "Acme", ← string inside the object
    "dbaName": "Acme"    ← string inside the object
  },
  "docType": "Account"   ← string (not an object, no braces)
}
What is properties?
properties is the keyword that lists all the named fields INSIDE an object.
Every field that belongs to an object is declared inside its properties block.
properties only appears when type is object — it is how you define what the object contains.

Structure:
{
  "type": "object",          ← declares this is an object (a container)
  "properties": {            ← opens the list of fields inside this object
    "fieldOne": {            ← first field definition
      "type": "string"       ← what type fieldOne holds
    },
    "fieldTwo": {            ← second field definition
      "type": "integer"      ← what type fieldTwo holds
    }
  }                          ← closes the properties list
}

In your schema the ROOT object has these properties:
  "properties": {
    "docType":              { type: string,  enum: [...] }
    "accountNumber":        { type: integer, format: int64 }
    "name":                 { type: object,  properties: { legalName, dbaName } }   ← object inside object!
    "address":              { type: object,  properties: { line1, city, state, countryCode } }
    "dob":                  { type: string,  pattern: "..." }
    "status":               { type: string,  enum: [...] }
    "customerId":           { type: integer, format: int64 }
    "lastUpdateDate":       { type: string,  format: date }
    "lastUpdateTimestamp":  { type: string,  format: date-time }
  }

The name object has its OWN properties nested inside:
  "name": {
    "type": "object",        ← name is itself an object
    "properties": {          ← these are name's own sub-fields
      "legalName": { "type": "string", "maxLength": 20 },
      "dbaName":   { "type": "string", "maxLength": 20 }
    }
  }

IMPORTANT: properties does NOT mean "required".
A field listed in properties is OPTIONAL by default.
To make a field required you must also list it in the required array.

  "required": ["docType", "name", "address", "status"]
  → Only these 4 must be present. All other properties are optional.
Object Nesting — Visualized in Your Schema
Your schema has 3 levels of nesting:

  Level 1 — Root Object (CustomerAccount)
  {
    "type": "object"          ← the whole record is an object
    "properties": {
      "docType":       string
      "accountNumber": integer
      "status":        string
      "customerId":    integer

      Level 2 — name Object (nested inside root)
      "name": {
        "type": "object"      ← name is also an object
        "properties": {
          Level 3 — simple fields inside name
          "legalName":  string  ← just a string, no more nesting
          "dbaName":    string  ← just a string, no more nesting
        }
      }

      Level 2 — address Object (nested inside root)
      "address": {
        "type": "object"      ← address is also an object
        "properties": {
          Level 3 — simple fields inside address
          "line1":       string
          "city":        string
          "state":       string  pattern: ^[A-Z]{2}$
          "countryCode": string  pattern: ^[A-Z]{3}$
        }
      }
    }
  }

When this schema validates an incoming API request it checks:
  → Is the root value a JSON object (has { } braces)?
  → Does it have the required fields: docType, name, address, status?
  → Is name itself an object with { } braces?
  → Does name.legalName exist (required) and is it a string under 20 chars?
  → Is address an object with line1 present?
  → Does address.state match the pattern ^[A-Z]{2}$?
  → And so on for every field in properties...

Couchbase — Core Concepts

Cluster · Node · Bucket · TTL · Architecture

Infrastructure — Cluster, Node, Bucket

Physical Architecture
Couchbase Infrastructure Hierarchy
CLUSTER — The entire Couchbase deployment
  A Cluster is a group of Nodes (servers) working together as one system.
  All nodes share the same data and workload. The cluster appears as a
  single endpoint to your Spring Boot application — you connect to the
  cluster, not to individual nodes.

  Example:  couchbase-cluster.company.com  ← single connection string
                     |
         ┌───────────┼───────────┐
         ▼           ▼           ▼
      Node 1      Node 2      Node 3     ← 3-node cluster

NODE — A single server (machine) inside the cluster
  Each Node holds a portion of the data called vBuckets (virtual partitions).
  Couchbase has 1024 vBuckets distributed across all nodes.
  If you have 3 nodes, each node holds approximately 341 vBuckets.
  Nodes also replicate data to each other for fault tolerance.

  Node services:
    Data Service    → stores and retrieves documents
    Index Service   → maintains indexes for N1QL queries
    Query Service   → executes N1QL query engine
    Search Service  → full-text search capability

BUCKET — A logical container for documents (like a database)
  A Bucket is a named grouping of documents within the cluster.
  You can have multiple buckets in one cluster.
  Documents in different buckets are completely isolated from each other.

  Your schema uses:   customer_bucket
    → stores both Account and Customer documents (distinguished by doc_type)
    → has its own memory quota, replica count, and TTL settings

SCOPE — A namespace inside a bucket (like a schema in SQL)
  Introduced in Couchbase 7.0 for multi-tenancy inside a bucket.
  Example:  customer_bucket → retail_scope → account_collection

COLLECTION — A sub-grouping inside a scope (like a table in SQL)
  Documents of the same type live in a collection.
  Example:  retail_scope → accounts  (Account documents)
            retail_scope → customers (Customer documents)

FULL HIERARCHY:
  Cluster
    └── Bucket (customer_bucket)
          └── Scope (retail_scope)
                ├── Collection (accounts)
                └── Collection (customers)

TTL (Time To Live) — Document Expiry

Bucket Level · Document Level · Spring Boot
What is TTL?
TTL (Time To Live) tells Couchbase to automatically
delete a document after a set period of time.
Couchbase checks expiry lazily — expired documents
are removed during the next access or by a background
compaction process.

TTL can be set at two levels:

1. BUCKET LEVEL — applies to ALL documents in bucket
   Set in Couchbase Admin UI or CLI
   Every document expires after this time unless
   the document sets its own TTL

2. DOCUMENT LEVEL — set per individual document
   Overrides bucket-level TTL for that document
   Set in your Spring Boot code when saving

Common use cases for TTL in your C360 schema:
  → Cache documents that mirror Valkey (short TTL)
  → Session records that auto-expire
  → Temporary processing documents
  → Audit records with retention policy
Setting TTL in Spring Boot + Couchbase SDK
// Option 1 — Set TTL via @Document annotation
// Document expires after 3600 seconds (1 hour)
@Document(expiry = 3600)
public class CustomerAccount { ... }

// Option 2 — Set TTL programmatically per document
// Useful when different docs need different TTLs
@Autowired
private CouchbaseTemplate couchbaseTemplate;

public void saveWithTTL(
    CustomerAccount account, int ttlSeconds) {

  couchbaseTemplate.upsertById(
    CustomerAccount.class)
    .withExpiry(Duration.ofSeconds(ttlSeconds))
    .one(account);
}

// Option 3 — Raw SDK with expiry
collection.upsert(
  key,
  document,
  UpsertOptions.upsertOptions()
    .expiry(Duration.ofHours(24))
);

// Option 4 — Touch to reset TTL on existing doc
collection.touch(
  key,
  Duration.ofHours(24)
);

// Option 5 — Get and reset TTL simultaneously
collection.getAndTouch(
  key,
  Duration.ofHours(24)
);
Bucket-Level TTL — Admin CLI Setup
# Set bucket-level TTL of 86400 seconds (24 hours) via Couchbase CLI
couchbase-cli bucket-edit   --cluster localhost:8091   --username admin --password password   --bucket customer_bucket   --max-ttl 86400

# Or via REST API
curl -X POST http://localhost:8091/pools/default/buckets/customer_bucket   -u admin:password   -d maxTTL=86400

# TTL values:
#   0       = documents never expire (default — permanent storage)
#   3600    = 1 hour
#   86400   = 24 hours
#   604800  = 7 days
#   2592000 = 30 days

Document Key Design

Best Practices for your C360 Schema
Key Naming Conventions
Couchbase document keys are strings. Best practice is to use a prefix
that identifies the document type so keys are globally unique across buckets.

For your schema:
  Account documents:   ACCOUNT::123456789    (account_number as key)
  Customer documents:  CUSTOMER::987654321   (customer_id as key)

Benefits of prefix pattern:
  → Prevents key collision if account_number and customer_id share same value
  → Makes the key self-describing — you know the doc type from the key alone
  → Easier to manage and debug in Couchbase UI

In Spring Boot:
public String buildKey(CustomerAccount account) {
  if ("Account".equals(account.getDocType())) {
    return "ACCOUNT::" + account.getAccountNumber();
  }
  return "CUSTOMER::" + account.getCustomerId();
}

Elasticsearch — Core Concepts

Cluster · Node · Index · Mapping · keyword vs text · copy_to

Infrastructure — Cluster, Node, Index, Shard

Physical Architecture
Elasticsearch Infrastructure Hierarchy
CLUSTER — The entire Elasticsearch deployment
  A Cluster is a group of Nodes. All nodes work together as one search engine.
  Your Spring Boot application connects to the cluster via a single URL.
  Example:  http://elasticsearch-cluster.company.com:9200

NODE — A single server (machine) inside the cluster
  Each Node holds some Shards of data. Nodes come in different roles:
    Master Node   → manages cluster state, tracks which shards go where
    Data Node     → stores shards and handles search/indexing operations
    Ingest Node   → pre-processes documents before indexing
    Coordinating  → routes requests, merges results (every node can do this)

INDEX — A logical container for documents (like a database table)
  Your schema uses index:  customer_account
  An index holds all Account and Customer documents.
  Each index has its own Mapping (schema definition) and Settings.

SHARD — A physical slice of an index stored on a node
  Elasticsearch splits each index into multiple Shards for scalability.
  Primary Shards  → the main copy of the data
  Replica Shards  → backup copies stored on different nodes for redundancy
  Example with 3 nodes, 3 primary shards, 1 replica each:

                     Node 1          Node 2          Node 3
                  ┌──────────┐    ┌──────────┐    ┌──────────┐
  Primary Shards  │  P1  P2  │    │    P3    │    │          │
  Replica Shards  │          │    │  R1      │    │  R2  R3  │
                  └──────────┘    └──────────┘    └──────────┘

  If Node 2 fails → R1 (replica of P1) on Node 3 is promoted to primary.
  Data is never lost as long as at least one replica exists on a healthy node.

DOCUMENT — A single JSON record stored in an index
  Equivalent to one row in a database or one document in Couchbase.
  Each document has an _id field (like Couchbase document key).
  In your schema, _id = account_number or customer_id.

What is a Mapping?

Schema Definition for Elasticsearch
Mapping Explained
A Mapping is Elasticsearch's schema definition file.
It tells Elasticsearch:
  → What fields exist in your documents
  → What data type each field is (text, keyword, long, date, object...)
  → How each field should be indexed and searched
  → Which fields should be copied into combined search fields

CRITICAL: Mappings are defined BEFORE you insert data.
Once data exists, changing field types requires reindexing (recreating the index).
This is different from Couchbase which is fully schemaless.

Example — without a mapping, Elasticsearch auto-detects types (dynamic mapping):
  "status": "ACTIVE"        → Elasticsearch guesses type: text  (WRONG for our use)
  "account_number": 123456  → Elasticsearch guesses type: long  (correct)
  "last_update_date": "2026-03-01" → guesses type: text (WRONG — should be date)

With our explicit mapping:
  "status"            → keyword   (exact match only — correct)
  "account_number"    → long      (number — correct)
  "last_update_date"  → date with format "yyyy-MM-dd" (correct, enables range queries)
  "legal_name"        → text + keyword multi-field (both full-text and exact — correct)

keyword vs text — The Most Important Elasticsearch Concept

Exact Match vs Full-Text Search
keyword — Exact Match
keyword stores the value exactly as-is.
It does NOT break the value into words.
Use for: filtering, sorting, aggregating,
         exact value matching.

When you index this document:
{
  "status": "ACTIVE",
  "country_code": "USA",
  "state": "TX"
}

Elasticsearch stores exact strings.
Queries that WORK with keyword:
{ "term": { "status": "ACTIVE" }}    ✓ works
{ "term": { "state": "TX" }}         ✓ works

Queries that FAIL with keyword:
{ "term": { "status": "active" }}    ✗ fails (case sensitive)
{ "match": { "status": "ACT" }}      ✗ fails (no partial match)

Fields using keyword in our schema:
  doc_type, status, state,
  country_code, dob
text — Full-Text Search
text breaks the value into individual tokens
(words) using an analyzer, then indexes each
token separately.
Use for: searching within content, partial
         word matching, relevance ranking.

When you index this document:
{
  "legal_name": "Acme Corporation"
}

Standard analyzer tokenizes it as:
  ["acme", "corporation"]
  (lowercased, split on spaces)

Queries that WORK with text:
{ "match": { "legal_name": "Acme" }}      ✓ works
{ "match": { "legal_name": "acme" }}      ✓ works (case insensitive)
{ "match": { "legal_name": "corporation" }}✓ works

Queries that FAIL with text:
{ "term": { "legal_name": "Acme Corporation" }}✗ fails
(term query needs exact — use .keyword for that)

Fields using text in our schema:
  legal_name, dba_name, line1, city,
  full_text_srch
Multi-Field — text AND keyword on the Same Field
Our legal_name and dba_name use BOTH types via the fields (multi-field) feature.
This lets you do both full-text AND exact match on the same field.

Mapping definition:
"legal_name": {
  "type": "text",               ← main field is text (full-text search)
  "analyzer": "standard",
  "fields": {
    "keyword": {
      "type": "keyword",        ← sub-field for exact match
      "ignore_above": 20
    }
  }
}

How to query each sub-type:
  Full-text search:   { "match": { "name.legal_name": "Acme" }}               → finds "Acme Corporation"
  Exact match:        { "term":  { "name.legal_name.keyword": "Acme Corporation" }} → exact only
  Sort alphabetically: { "sort": [{ "name.legal_name.keyword": "asc" }]}      → must use .keyword for sort

copy_to — The Combined Search Field Explained

full_text_srch — How It Works
copy_to Explained Step by Step
copy_to is an Elasticsearch feature that copies the value of one field
into another field at index time. The destination field is virtual —
it exists ONLY inside Elasticsearch for search purposes.
It is NOT stored in the document and NOT returned in API responses.

In our mapping, 4 fields copy their values into full_text_srch:
  name.legal_name  ─┐
  name.dba_name    ─┤
  address.line1    ─┤──► full_text_srch  (combined search field)
  address.city     ─┘

When this document is indexed:
{
  "name": { "legal_name": "Acme Corporation", "dba_name": "Acme Corp" },
  "address": { "line1": "123 Main Street", "city": "Dallas" }
}

Elasticsearch internally creates a full_text_srch field containing:
  "Acme Corporation Acme Corp 123 Main Street Dallas"
  (all 4 field values concatenated and tokenized together)

Now a single query searches all 4 fields at once:
  { "match": { "full_text_srch": "Acme Dallas" }}
  → finds the document because both "Acme" and "Dallas" appear in full_text_srch

Without copy_to you would need a multi_match query instead:
  { "multi_match": {
    "query": "Acme Dallas",
    "fields": ["name.legal_name", "name.dba_name", "address.line1", "address.city"]
  }}

copy_to is more efficient — the combined index is pre-built at write time,
not computed at query time, so searches on full_text_srch are faster.

DynamoDB — Core Concepts

Partition · PK · SK · GSI · Single Table Design · Billing

How DynamoDB Stores Data — Partitions

Physical Storage Model
Partition Explained
PARTITION — A physical storage unit inside DynamoDB
  DynamoDB is fully managed — you never see servers, nodes, or clusters.
  AWS handles all infrastructure. Internally, DynamoDB splits table data
  into Partitions across many physical servers.

  Each partition:
    → Stores up to 10 GB of data
    → Handles up to 3,000 read capacity units per second
    → Handles up to 1,000 write capacity units per second
    → Lives on AWS-managed SSDs in multiple availability zones

HOW PARTITIONING WORKS:
  DynamoDB runs your PK (Partition Key) through a hash function.
  The hash result determines which physical partition holds that record.

  PK = "ACCOUNT#123456789"
         ↓
  Hash Function  →  hash value 847
         ↓
  Partition 3   (holds records with hash range 800-900)

  This is why PK is called "Partition Key" — it determines the partition.

WHY THIS MATTERS FOR DESIGN:
  All records with the same PK live on the SAME partition.
  If you make all records have PK = "ALL_ACCOUNTS" every read and write
  hits one partition — this is called a HOT PARTITION and kills performance.
  Good PK design spreads data evenly across many partitions.

  Good PK:   ACCOUNT#123456789 (unique per account — spread evenly)
  Bad PK:    "Account"         (all accounts on one partition — HOT)

PK and SK — Partition Key and Sort Key

Primary Key Design
PK — Partition Key (HASH)
PK is the first part of the primary key.
It determines WHICH PARTITION stores the record.
Think of it as the filing DRAWER.

  PK = "ACCOUNT#123456789"

Rules for PK:
  → Must be UNIQUE per record (if no SK)
  → If SK exists, PK can repeat — all records
    with same PK live in the same partition
    and are sorted by SK
  → DynamoDB can only filter by PK using
    equality (=) — no range queries on PK
  → Type must be String (S), Number (N),
    or Binary (B)

In our schema PK design:
  Account:   pk = "ACCOUNT#123456789"
  Customer:  pk = "CUSTOMER#987654321"

KeyType: HASH means Partition Key
  { "AttributeName": "pk",
    "KeyType": "HASH" }
SK — Sort Key (RANGE)
SK is the second part of the primary key.
It determines ORDER within a partition.
Think of it as the order of files INSIDE the drawer.

  SK = "METADATA#123456789"

Rules for SK:
  → Multiple records can share the same PK
    as long as each has a different SK
  → DynamoDB can query SK using:
    =  begins_with  between  >  >=  <  <=
  → Enables powerful range queries within
    a partition

Power of PK + SK together:
  All records for one account:
  PK = "ACCOUNT#123456789"
    SK = "METADATA#123456789"  → account record
    SK = "ADDRESS#001"         → address record
    SK = "AUDIT#2026-03-01"    → audit entry
    SK = "AUDIT#2026-03-02"    → audit entry

  One query fetches ALL related records:
  pk = "ACCOUNT#123456789"
  sk begins_with "AUDIT#"
  → returns all audit entries for that account

KeyType: RANGE means Sort Key
  { "AttributeName": "sk",
    "KeyType": "RANGE" }

GSI — Global Secondary Index

Querying Non-Key Fields
Why GSI Exists and How It Works
PROBLEM: DynamoDB can ONLY query by PK and SK directly.
If you want to find all ACTIVE accounts you cannot do:
  SELECT * FROM table WHERE status = "ACTIVE"
  This requires a full table scan — extremely slow and expensive.

SOLUTION: Global Secondary Index (GSI)
  A GSI is a complete copy of the table reorganized around a different key.
  AWS maintains the GSI automatically — every write to the main table
  is automatically propagated to all GSIs.

  Main table:        PK = "ACCOUNT#123456789"  SK = "METADATA#123456789"
  status-index GSI:  PK = "ACTIVE"             (points to same document)

Our schema has 2 GSIs:

  status-index
    GSI PK = status
    Query: "give me all ACTIVE accounts"
    { "IndexName": "status-index",
      "KeySchema": [{ "AttributeName": "status", "KeyType": "HASH" }] }

  customer_id-index
    GSI PK = customer_id
    Query: "give me the account linked to customer 987654321"
    { "IndexName": "customer_id-index",
      "KeySchema": [{ "AttributeName": "customer_id", "KeyType": "HASH" }] }

Projection: ALL means the GSI includes ALL fields from the main table.
Other options: KEYS_ONLY (just PK/SK), INCLUDE (specific fields only)

LSI (Local Secondary Index) — alternative to GSI:
  LSI must be created at table creation time (GSI can be added later)
  LSI uses the same PK as main table but different SK
  LSI is limited to 10 GB per PK value
  For most use cases GSI is more flexible and preferred

AttributeDefinitions · Data Types · BillingMode

DynamoDB Table Settings
AttributeDefinitions — Key Fields Only
DynamoDB only requires type declarations
for KEY fields (PK, SK, and GSI keys).
All other fields are schemaless — send
any field and DynamoDB stores it.

Only 3 attribute types exist in DynamoDB:
  S  = String    "ACTIVE", "Account", "USA"
  N  = Number    123456789, 3.14
  B  = Binary    encrypted bytes, images

Our key fields are all Strings (S):
"AttributeDefinitions": [
  { "AttributeName": "pk",
    "AttributeType": "S" },    ← String
  { "AttributeName": "sk",
    "AttributeType": "S" }     ← String
]

Note: account_number is a Number but our
PK is "ACCOUNT#123456789" (String with prefix)
so AttributeType is S, not N.
BillingMode — PAY_PER_REQUEST vs PROVISIONED
PROVISIONED — You declare capacity upfront:
  ReadCapacityUnits:  how many reads/sec
  WriteCapacityUnits: how many writes/sec
  AWS charges for declared capacity whether
  you use it or not.
  Good for: predictable, steady workloads

  "BillingMode": "PROVISIONED",
  "ProvisionedThroughput": {
    "ReadCapacityUnits": 100,
    "WriteCapacityUnits": 50
  }

PAY_PER_REQUEST — Serverless, auto-scales:
  No capacity planning needed.
  DynamoDB scales instantly to any traffic.
  You pay only for actual reads and writes.
  Good for: unpredictable or spiky traffic
  Perfect for Project Diamond scaling from
  60M to 300M requests/day.

  "BillingMode": "PAY_PER_REQUEST"

For C360 at 300M requests/day:
  → PAY_PER_REQUEST is recommended
  → No risk of throttling during traffic spikes
  → No capacity planning overhead

// Store in Valkey cache
public void cacheAccount(
    CustomerAccount account) {

  String key = "ACCOUNT:"
    + account.getAccountNumber();
  HashOperations<String,String,String>
    hashOps = redisTemplate.opsForHash();

  hashOps.put(key, "doc_type",
    account.getDocType());
  hashOps.put(key, "account_number",
    account.getAccountNumber().toString());
  hashOps.put(key, "name:legal_name",
    account.getName().getLegalName());
  hashOps.put(key, "status",
    account.getStatus());

  // Expire after 1 hour
  redisTemplate.expire(key,
    Duration.ofHours(1));
}

// Retrieve from Valkey
public Map<String,String> getAccount(
    Long accountNumber) {
  String key = "ACCOUNT:" + accountNumber;
  return redisTemplate
    .opsForHash().entries(key);
}

@RestController
@RequestMapping("/api/v1/accounts")
public class CustomerAccountController {

  @Autowired
  private CustomerAccountService service;

  // ─── GET by Account Number ───────────────────────────────────────
  @GetMapping("/{accountNumber}")
  public ResponseEntity<CustomerAccount> getByAccountNumber(
      @PathVariable Long accountNumber) {

    CustomerAccount account = service.findByAccountNumber(accountNumber);
    if (account == null) {
      return ResponseEntity.notFound().build();
    }
    return ResponseEntity.ok(account);  // Returns camelCase JSON
  }

  // ─── POST — Create New Account ───────────────────────────────────
  @PostMapping
  public ResponseEntity<CustomerAccount> createAccount(
      @RequestBody @Valid CustomerAccount account) {

    account.setLastUpdateTimestamp(
      Instant.now().toString());
    account.setLastUpdateDate(
      LocalDate.now().toString());

    CustomerAccount saved = service.save(account);

Service Layer — Business Logic + Multi-Datastore Writes

@Service · Couchbase + Elasticsearch + Valkey
CustomerAccountService.java
@Service
public class CustomerAccountService {

  @Autowired private CouchbaseRepository couchbaseRepo;
  @Autowired private ElasticsearchClient esClient;
  @Autowired private RedisTemplate<String, String> redisTemplate;

  // ─── SAVE — Writes to Couchbase + Elasticsearch + Valkey Cache ───
  public CustomerAccount save(CustomerAccount account) {

    // 1. Save to Couchbase (primary store)
    couchbaseRepo.save(account);

    // 2. Index in Elasticsearch (for search)
    esClient.index(i -> i
      .index("customer_account")
      .id(account.getAccountNumber().toString())
      .document(account));

    // 3. Cache in Valkey (for fast reads)
    cacheInValkey(account);

    return account;
  }

  // ─── FIND — Cache-first, fallback to Couchbase ────────────────────
  public CustomerAccount findByAccountNumber(Long accountNumber) {

    String cacheKey = "ACCOUNT:" + accountNumber;

    // 1. Check Valkey cache first
    Map<Object, Object> cached =
      redisTemplate.opsForHash().entries(cacheKey);

    if (!cached.isEmpty()) {
      return mapToAccount(cached);  // Cache hit — return immediately
    }

    // 2. Cache miss — fetch from Couchbase
    CustomerAccount account =
      couchbaseRepo.findById(accountNumber).orElse(null);

    // 3. Populate cache for next request
    if (account != null) {
      cacheInValkey(account);
    }

    return account;
  }

  // ─── SEARCH BY STATUS — via Couchbase N1QL ────────────────────────
  public List<CustomerAccount> findByStatus(String status) {
    return couchbaseRepo.findByStatus(status);
  }

  // ─── FULL-TEXT SEARCH — via Elasticsearch ─────────────────────────
  public List<CustomerAccount> fullTextSearch(
      String query, int page, int size) {

    SearchResponse<CustomerAccount> response = esClient.search(s -> s
      .index("customer_account")
      .from(page * size)
      .size(size)
      .query(q -> q
        .match(m -> m
          .field("full_text_srch")
          .query(query))),
      CustomerAccount.class);

    return response.hits().hits().stream()
      .map(Hit::source)
      .collect(Collectors.toList());
  }

  // ─── DATE RANGE SEARCH — via Elasticsearch ────────────────────────
  public List<CustomerAccount> findByTimestampRange(
      String from, String to) {

    SearchResponse<CustomerAccount> response = esClient.search(s -> s
      .index("customer_account")
      .query(q -> q
        .range(r -> r
          .field("last_update_timestamp")
          .gte(JsonData.of(from))
          .lte(JsonData.of(to)))),
      CustomerAccount.class);

    return response.hits().hits().stream()
      .map(Hit::source)
      .collect(Collectors.toList());
  }

  // ─── CACHE HELPER — Write to Valkey Hash ──────────────────────────
  private void cacheInValkey(CustomerAccount account) {
    String key = "ACCOUNT:" + account.getAccountNumber();
    HashOperations<String, String, String> hashOps =
      redisTemplate.opsForHash();

    hashOps.put(key, "doc_type",       account.getDocType());
    hashOps.put(key, "account_number", account.getAccountNumber().toString());
    hashOps.put(key, "status",         account.getStatus());
    hashOps.put(key, "name:legal_name",account.getName().getLegalName());
    hashOps.put(key, "address:city",   account.getAddress().getCity());
    hashOps.put(key, "last_update_timestamp",
                                         account.getLastUpdateTimestamp());
    redisTemplate.expire(key, Duration.ofHours(1));
  }
}

Complete Request Flow — POST an Account

End-to-End · All Layers
What happens when POST /api/v1/accounts is called
1. Client sends camelCase JSON to Spring Boot API
   → { "accountNumber": 123456789, "name": { "legalName": "Acme" }, "status": "ACTIVE" }

2. @Valid runs bean validation — checks @Pattern, @Size, @NotBlank rules
   → If invalid → 400 Bad Request with field-level error messages returned immediately

3. @RequestBody + Jackson deserializes camelCase JSON → Java CustomerAccount object
   → accountNumber = 123456789, legalName = "Acme" (Java camelCase fields)

4. Service layer sets lastUpdateTimestamp = Instant.now()

5. @JsonNaming(SnakeCaseStrategy) serializes Java object → snake_case JSON for Couchbase
   → { "account_number": 123456789, "name": { "legal_name": "Acme" }, "status": "ACTIVE" }

6. Couchbase saves document with key = "123456789"

7. Elasticsearch indexes the same document in customer_account index
   → legal_name + dba_name + line1 + city copied into full_text_srch field automatically

8. Valkey caches the record as Hash with key ACCOUNT:123456789 + 1 hour TTL

9. Spring Boot serializes Java object back to camelCase JSON for API response
   → { "accountNumber": 123456789, "name": { "legalName": "Acme" }, "status": "ACTIVE" }

10. Client receives 201 Created with camelCase response body

─────────────────────────────────────────────────────────────────────────────
  Client (camelCase) → Spring Boot → Couchbase (snake_case)
                                   → Elasticsearch (snake_case)
                                   → Valkey Cache (snake_case hash)
                     ← Spring Boot ← Response (camelCase)
